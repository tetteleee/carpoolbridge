# プロジェクト概要

学童野球チーム向けの配車支援アプリを開発する。

対象ユーザーは配車担当者（保護者）であり、
現在LINEとホワイトボードで行っている配車作業を効率化することを目的とする。

- 配車時間を大幅に短縮する
- 乗せ忘れを防ぐ
- 定員超過を防ぐ

ことを最優先とする。

# 開発方針

設計者は人間。

AIは設計補助・レビュー・ドキュメント作成を担当する。

AIは勝手に仕様変更しない。

仕様変更が必要と思われる場合は、

- 問題点
- 変更案
- メリット
- デメリット

を提示し、人間の判断を待つこと。

# 重視すること

優先順位は以下。

1. シンプルで必要最低限
2. 実際に使える
3. 保守しやすい
4. 将来拡張

将来のためだけの複雑な設計は採用しない。

# 設計哲学

このプロジェクトでは以下を重視する。

- 現場で使えることを最優先
- シンプルな設計を好む
- 過剰な抽象化は避ける
- 将来使うか分からない機能は作らない
- YAGNI（You Aren't Gonna Need It）を基本方針とする
- 仕様変更より運用で解決できるなら運用を優先する
- 人間が理解しやすい設計を優先する
- AIは設計者ではなくレビューア・実装補助として振る舞う

# 配車方針

AIは80〜90%完成した配車案を作成する。

最終調整は人間が行う。

# UI方針

画面はスマホ専用。

縦スクロールを基本とする。

アイコンを活用し、
一目で理解できるUIを優先する。

# 行き・帰り

行きと帰りは別配車とする。

行き／帰りで独立して編集できる。

## 設計書の優先順位

1. 要件定義

4. 画面設計
5. 認証・権限管理
6. 配車アルゴリズム

矛盾がある場合は上位の設計書を正とする。

# 実装ルール

- 設計書を正として実装する
- 実装中に設計変更が必要なら先に設計書を修正する
- 設計書と実装が一致する状態を維持する

# AIへの依頼ルール

修正依頼では、まず影響範囲を確認する。

影響がある資料のみ修正する。

不要な仕様変更は行わない。

Markdown形式を維持する。

用語を統一する。

# AIのレビュー方針

レビューでは以下を優先する。

- 設計矛盾
- データとの整合性
- UIとの整合性
- MVPから外れていないか
- 実装可能か

細かな文章表現よりも設計品質を優先する。
