# Task008 ドラッグ＆ドロップ

Version: 1.1

---

# 目的

05_画面設計.md 8.2.1の一覧表示（複数車を横スクロールで俯瞰する画面）で、
子供・コーチのチップを長押し＋ドラッグして別の車へ移動できるようにする。

あわせて、直前の修正を複数手戻せる「戻す」（Undo）機能を実装する。

UC-06「配車を修正する」のうち、ドライバー変更・巡回順変更・その他の
乗車者の手動追加は本Taskに含めない（背景章参照、別Task008aとして
切り出す）。

---

# 背景

Task007（配車画面）で、配車結果の一覧表示・詳細表示・「+ 車を追加」までは
実装済みという前提で本Taskを進める。

05_画面設計.md 8章では、子供チップの操作を以下のように使い分けると
定義されている。

- タップ＝AI判断理由を表示（8.3、Task007で実装済み）
- 長押し＋ドラッグ＝車の移動（本Task）

また、同ドキュメントでは「詳細ビュー（1台のみ表示）を開いている間は、
他の車が画面上に無いためドラッグ移動はできない仕様とする」とされている。
本Taskでもこの制約を踏襲し、**ドラッグ移動は一覧表示（8.2.1）でのみ**
可能とする。

UC-06には他に「ドライバー変更」「巡回順変更」「その他の乗車者の
手動追加」という操作も定義されているが、これらはドラッグ操作と
直接関係しない別UIになるため、別Task（Task008a、本Task内では
実装しない）とする。

---

# 前提とする決定事項

- ドラッグ実装には `@dnd-kit/core` を使用する。長押し起動（PointerSensor +
  activation constraint の delay指定）でタップ操作と競合しない実装が
  比較的容易なため採用する。タッチ操作を独自実装する場合より
  実装・検証コストが低いと判断した（04_開発方針.mdの「不要なライブラリは
  追加しない」原則の例外として、本Task固有の理由をここに明記する）
- 「戻す」は複数手戻せる履歴スタック方式とする。スタックは
  DispatchPageのReact state内にのみ保持し、画面リロードで消えてよい
  （Firestoreへの永続化は行わない）
- 定員超過・兄弟分離・集合場所分割の警告は、ドラッグ移動の直接的な
  結果として表示する（UC-06のシステム要件）。警告はモーダルで
  ブロックせず、移動自体は許可した上で車カードに警告表示を行う
  （現場の最終判断を優先する、00_開発方針.mdの基本方針に準拠）
- チップを長押しでドラッグ開始した瞬間に、Web Vibration API
  （`navigator.vibrate()`）で短い振動フィードバックを発生させる。
  iOS Safariは同APIに対応していないため振動しない
  （エラーにはならず、単に無視される）。Android Chromeでは振動する。
  この非対応差はブラウザ側の制約であり、本Taskでは代替手段
  （視覚的なフィードバック強化等）を追加しない

---

# 完了条件

- 一覧表示（8.2.1）で、子供・コーチのチップを長押しするとドラッグ可能
  になり、別の車のカード、または「未配車」エリアへドロップできる
- ドロップすると、該当のCarpool.membersが更新され、Firestoreへ
  即時反映される
- タップ操作（AI判断理由の表示、Task007で実装済み）は、長押し＋ドラッグと
  誤操作なく使い分けられる
- 長押しでドラッグが開始された瞬間に、対応端末（Android等）では
  短い振動フィードバックが発生する
- 詳細ビュー（8.2.2）ではドラッグ操作ができない（Task007の仕様を維持）
- 移動の結果、以下に該当する場合は該当する車カードに警告が表示される
  - 定員超過（乗車人数が`capacity`を超える）
  - 兄弟分離（同じfamilyIdの子供が別の車に分かれている）
  - 集合場所分割（同じ集合場所の子供が、必要以上に複数の車に
    分かれている。7章参照）
- 画面左上に「戻す」ボタンが常設され、押すたびに直前の移動を
  1手ずつ取り消せる（複数回押せば複数手戻せる）
- 「戻す」を押すと、Firestore上のCarpoolデータも巻き戻った状態に
  更新される
- npm run build が成功する

# 実装内容

## 1. ライブラリ導入

`@dnd-kit/core` を導入する（前提1参照）。

```
npm install @dnd-kit/core
```

---

## 2. ドラッグ可能なチップの実装

Task007で実装済みの子供・コーチチップ（一覧表示内）を
`useDraggable`でラップする。

```typescript
import { useDraggable } from "@dnd-kit/core";

// PointerSensorのactivationConstraintでdelayを指定し、
// 長押し後にドラッグが開始されるようにする（DndContext側で設定）
```

タップ（クリック）とドラッグ開始が競合しないよう、
`DndContext`の`sensors`に`PointerSensor`を`activationConstraint: { delay: 200, tolerance: 5 }`
程度の設定で登録する（数値は実装時に操作感を見ながら調整してよい）。

`DndContext`の`onDragStart`で、短い振動フィードバックを発生させる。

```typescript
function handleDragStart() {
  if ("vibrate" in navigator) {
    navigator.vibrate(15); // 15ms程度の短い振動（数値は実装時に調整してよい）
  }
  // ...ドラッグ開始時の既存処理
}
```

iOS Safariには`navigator.vibrate`自体が存在しないため、
`"vibrate" in navigator`がfalseとなり何も起きない
（例外は発生しない）。

---

## 3. ドロップ先（車カード・未配車エリア）の実装

各車カード・未配車エリアを`useDroppable`でラップし、
一意のID（CarpoolのID、または"unassigned"）を割り当てる。

---

## 4. ドロップ時の更新処理

`DndContext`の`onDragEnd`で、

1. 移動元のCarpool（または未配車リスト）からmemberを取り除く
2. 移動先のCarpool（または未配車リスト）へmemberを追加する
3. 更新前の状態を9章の履歴スタックへpushする
4. 更新後のCarpool一覧をFirestoreへ保存する（`replaceCarpools`、
   Task007で実装済みの関数を使う）
5. 6章の警告判定を再計算する

同一車内でのドロップ（移動なし）の場合は何もしない。

---

## 5. 詳細ビューでのドラッグ無効化

詳細ビュー（8.2.2、Task007で実装済み）表示中は、
`DndContext`を一覧表示側にのみ設置する（詳細ビューのメンバー行は
`useDraggable`でラップしない）ことで、ドラッグ不可を実現する。

---

## 6. 警告判定ロジック

`src/dispatch/warnings.ts` を新規作成する。

```typescript
export type CarpoolWarning =
  | "定員超過"
  | "兄弟分離"
  | "集合場所分割";

export function getCarpoolWarnings(
  carpool: Carpool,
  allCarpools: Carpool[],
  context: DispatchContext // Task007のcontextを再利用
): CarpoolWarning[] { ... }
```

- 定員超過：`members.length > capacity`
- 兄弟分離：`members`内の子供について、同じfamilyIdを持つ
  「参加している他の子供」が`allCarpools`内の別の車に存在する場合
- 集合場所分割：`members`内の子供のpickupLocationIdについて、
  同じpickupLocationIdを持つ「参加している他の子供」が
  `allCarpools`内の別の車に存在する場合
  （簡易判定でよく、意図的な分割か誤操作かの区別はしない。
  UC-06の「問題がある場合は警告を表示する」という位置づけの通り、
  最終判断は担当者に委ねる）

各車カードに、該当する警告があれば5章のCarpoolTypeの表示に加えて
警告バッジ（例：⚠️定員超過）を表示する。既存の「定員超過時は枠を赤く
する」表示（Task007実装済み）はそのまま活用してよい。

---

## 7. 履歴スタックと「戻す」ボタン

DispatchPageに以下の状態を追加する。

```typescript
const [history, setHistory] = useState<Carpool[][]>([]);
```

- ドロップ処理の直前（4章の手順3）で、現在のCarpool一覧を
  `history`へpushする
- 「戻す」ボタン押下時、`history`から最後の状態をpopし、
  現在のCarpool一覧として復元した上でFirestoreへ保存する
- `history`が空の場合、「戻す」ボタンは非活性にする
- 「配車作成」の再実行（Task007実装済み）や「+ 車を追加」
  （Task007実装済み）を行った場合は、履歴をクリアする
  （配車全体が作り直される操作のため、部分的な巻き戻しと
  整合しなくなるため）

---

# スコープ外

以下は実装しない。別Taskとして切り出す。

- ドライバー変更
- 巡回順（経由地の順序）変更
- 「その他の乗車者」（マスタに存在しない例外的な同乗者）の
  自由入力での手動追加・削除
- 「戻す」の逆（「やり直す」／Redo）
- 複数チップの同時選択・一括移動
- Firestoreへの履歴永続化（画面リロードで履歴が消えることを許容する）
- 集合場所分割・兄弟分離警告の「意図的な分割かどうか」の判定
  （常に警告として表示し、無視するかどうかは担当者の判断に委ねる）

# 確認項目

```
□ npm install / npm run build が成功する

□ 一覧表示で子供チップを長押しするとドラッグ状態になる

□ 対応端末（Android等）で、ドラッグ開始時に短い振動が発生する

□ iOS端末で振動が発生しなくてもエラーにならない

□ ドラッグしたチップを別の車カードへドロップすると、
  そのCarpoolのmembersが更新される

□ ドラッグしたチップを未配車エリアへドロップすると、
  未配車リストに戻る

□ 通常のタップ（長押しなし）ではAI判断理由が表示され、
  誤ってドラッグが開始されない

□ 詳細ビューを開いている間はドラッグができない

□ 定員を超える車に移動すると警告が表示される

□ 兄弟が別の車に分かれる移動をすると警告が表示される

□ 「戻す」ボタンで直前の移動が取り消される

□ 「戻す」を複数回押すと、複数手前の状態まで戻せる

□ 「配車作成」を再実行すると履歴がクリアされ、
  「戻す」ボタンが非活性になる

□ 移動結果がFirestoreへ反映され、リロードしても保持される

□ スマホ実機（またはブラウザのタッチエミュレーション）で
  長押し→ドラッグの操作感に違和感がないか確認する
```

# 実装ルール

- TypeScriptで実装する
- anyは使用しない
- 関数コンポーネント・Hooksで実装する
- React Hooksのルールを守る
- MVPなので過剰な抽象化はしない
- `@dnd-kit/core`以外の新規ライブラリは追加しない
