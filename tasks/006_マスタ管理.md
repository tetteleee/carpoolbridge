# Task006 マスタ管理

Version: 1.0

---

# 目的

UC-09〜UC-12（家庭・子供・集合場所・目的地）に対応するマスタ管理画面を実装し、
Task005で導入したFirebase接続を使って実際にFirestoreへ登録・編集・削除できる
ようにする。

本Taskは、モックストア／インメモリストアではなく実際にFirestoreへ
読み書きを行う最初のTaskとなる。

---

# 背景

05_画面設計.md 10章には「管理（暫定）」画面が既にワイヤーフレームとして
存在するが、そこに掲載されているのは集合場所・家庭・子供の3つのみで、
目的地の登録UIが明記されていない。目的地はUC-12でMVP対象と定義されて
いるため、本Taskで集合場所と同様の簡易フォームを追加する。

また、以下2点はドキュメント間の未確定事項のため、本Taskでは
以下の方針で実装する（実装ツールが独自判断しないよう、ここで確定させる）。

## コーチ管理について

04_データ設計.md 5章のCoach（コーチ）マスタには`familyId`（所属家庭、任意）
が追加されており、家庭に紐づくコーチは02_要件定義.md 8章・05_画面設計.md
7章の`coachParticipating`（コーチの参加要否）機能で使用される。
そのため、家庭にコーチが紐づいているかどうかを登録できる手段が
本Taskで必要になる。

一方、03_ユースケース.mdの「MVP対象」一覧には独立したコーチ管理画面
（UC-09〜12のような単独のCRUD画面）は含まれていない。

本Taskでは以下の方針とする。

- 独立した「コーチ管理」セクション（コーチ単体の一覧・編集画面）は
  実装しない
- 代わりに、4章の**家庭登録フォームに「コーチ名」欄（任意）を追加**する。
  コーチ名が入力された場合、`familyId`をその家庭のIDとした`Coach`
  ドキュメントを自動的に作成・更新する（詳細は4章参照）
- 家庭に紐づかない単独コーチ（`familyId`未設定）の登録・編集は
  本Taskでは対応しない。従来通りFirebase Consoleから直接登録する運用
  とする（06_認証・権限管理設計.mdのstaffUsers登録と同様の位置づけ）

コーチ単体の一覧・編集が必要になった場合は、別途Task化を検討する。

## 集合場所・目的地の削除方式について

04_データ設計.md 12章の論理削除方針（`isActive`）はFamily・Child・Coach
にのみ定義されており、PickupLocation・Destinationには`isActive`フィールドが
存在しない。一方でEvent（`destinationId`）・Child（`pickupLocationId`）が
これらを参照するため、無条件の物理削除は参照整合性を壊すリスクがある。

本Taskでは以下のルールとする。

- 集合場所・目的地は**物理削除**とする（`isActive`は追加しない）
- ただし、削除実行前に他ドキュメントから参照されていないかを確認し、
  **参照が1件でも存在する場合は削除ボタンを無効化し、理由を表示する**
  （4章参照）
- 家庭・子供は04_データ設計.md 12章の方針通り**論理削除（`isActive: false`）**
  とする

---

# 完了条件

- 「管理」画面（`/master`）から以下4種のマスタを登録・編集・削除できる
  - 集合場所（PickupLocation）
  - 家庭（Family）
  - 子供（Child）
  - 目的地（Destination）
- 家庭登録フォームで「コーチ名」（任意）を入力すると、対応する
  `Coach`ドキュメント（`familyId`=当該家庭のID）がFirestoreへ
  作成・更新される
- 登録・編集・削除はFirestoreへ即時反映される
- 家庭・子供の削除は論理削除（`isActive: false`）とし、一覧からは
  非表示になる
- 家庭を削除（`isActive: false`）すると、紐づく`Coach`が存在する場合は
  その`Coach`も連動して`isActive: false`になる
- 集合場所・目的地の削除は物理削除とするが、他ドキュメントから
  参照されている場合は削除できず、その旨がUIに表示される
- 子供登録時、所属家庭・集合場所を既存マスタからドロップダウンで
  選択できる
- npm run build が成功する

# 実装内容

## 1. 型定義

`src/types.ts` に以下を追加する（04_データ設計.md 3・4・6・7章準拠）。

```typescript
export interface Family {
  id: string;
  familyName: string;
  parentName: string;
  vehicleCapacity: number;
  isActive: boolean;
}

export interface Child {
  id: string;
  familyId: string;
  name: string;
  grade: number;
  pickupLocationId: string;
  isActive: boolean;
}

export interface PickupLocation {
  id: string;
  name: string;
  address: string;
  latitude: number;
  longitude: number;
}

export interface Coach {
  id: string;
  name: string;
  familyId: string | null; // 単独コーチはnull（本Taskでは扱わない）
  isActive: boolean;
}
```

`Destination` はTask003で定義済みのため、`address`フィールドが
不足していれば追加する（04_データ設計.md 7章準拠、`address`は任意）。

`createdAt`・`updatedAt`はFirestore側で`serverTimestamp()`を
使用して自動付与し、型定義上は本Taskでは省略可（一覧表示に使わないため）。

## 2. Firestoreアクセス関数

`src/firestore/families.ts`・`children.ts`・`pickupLocations.ts`・
`destinations.ts`・`coaches.ts` をそれぞれ新規作成し、以下のCRUD関数を実装する。

例（`families.ts`）

```typescript
export async function listFamilies(): Promise<Family[]> { ... }

// coachName が入力されていれば、familyId=このFamilyのidとしたCoachを
// 作成する（coaches.tsのupsertCoachForFamilyを呼び出す）
export async function addFamily(
  input: Omit<Family, "id" | "isActive">,
  coachName: string | null
): Promise<string> { ... }

// coachNameが空→紐づくCoachをisActive:falseに、
// coachNameが変更→既存Coachのnameを更新、
// 新規入力→Coachを新規作成する
export async function updateFamily(
  id: string,
  input: Partial<Family>,
  coachName: string | null
): Promise<void> { ... }

// isActive:falseにする。紐づくCoachが存在すれば、そのCoachも
// isActive:falseにする（coaches.tsのdeactivateCoachesByFamilyIdを呼ぶ）
export async function deactivateFamily(id: string): Promise<void> { ... }
```

`pickupLocations.ts`・`destinations.ts` は論理削除ではないため、
`deleteX` 関数とする（物理削除）。

`coaches.ts`（家庭紐づきコーチの最小限のCRUD、単独コーチは扱わない）

```typescript
// familyIdに紐づくCoachを1件取得する（家庭編集フォームの初期表示用）
export async function getCoachByFamilyId(familyId: string): Promise<Coach | null> { ... }

// familyIdに紐づくCoachが無ければ作成、あれば名前を更新する
export async function upsertCoachForFamily(familyId: string, name: string): Promise<void> { ... }

// familyIdに紐づくCoachをisActive:falseにする
export async function deactivateCoachesByFamilyId(familyId: string): Promise<void> { ... }
```

コレクション名は04_データ設計.md 11章に準拠する
（`families`・`children`・`pickupLocations`・`destinations`）。

## 3. 参照チェック関数

`src/firestore/referenceCheck.ts` を新規作成する。

```typescript
// 指定した集合場所IDを参照している子供が1件でも存在するか
export async function isPickupLocationInUse(locationId: string): Promise<boolean>

// 指定した目的地IDを参照しているイベントが1件でも存在するか
export async function isDestinationInUse(destinationId: string): Promise<boolean>
```

削除ボタン押下前にこれらを呼び出し、`true`の場合は削除を中止し
「他のデータで使用されているため削除できません」等のメッセージを表示する。

## 4. MasterPage の実装

`src/pages/MasterPage.tsx`（既存の暫定実装があれば置き換え、
なければ新規作成）を実装する。

05_画面設計.md 10章のワイヤーフレームに準拠し、以下の4セクションを
この順序で並べる（子供登録が家庭・集合場所に依存するため、
登録フォームもこの順に並べる）。

```
管理
──────────────────
集合場所を登録
 名称 [__________]
 住所 [__________]
 緯度 [______]
 経度 [______]
      [登録]
──────────────────
家庭を登録
 家庭名 [__________]
 保護者名 [__________]
 通常乗車人数 [__]
 コーチ名（任意） [__________]
      [登録]
──────────────────
子供を登録
 名前 [__________]
 学年 ▼小1〜小6
 所属家庭 ▼(登録済み家庭)
 集合場所 ▼(登録済み場所)
      [登録]
──────────────────
目的地を登録
 名称 [__________]
 住所 [__________]（任意）
 緯度 [______]
 経度 [______]
      [登録]
──────────────────
登録済み一覧
 各マスタごとにリスト表示
 各行に [編集] [削除] ボタン
```

- 各セクションの「登録済み一覧」に編集・削除ボタンを設ける
- 編集は同フォームにデータを読み込んで上書き保存する形でよい
  （モーダル等は不要、05_画面設計.md 10章の「使いやすさより
  登録できることを優先した暫定UI」という位置づけを踏襲する）
- 家庭・子供一覧には`isActive: false`のものは表示しない
  （04_データ設計.md 13章：非表示扱いだが検索は別途可能、
  ただし本Taskでは復活UIまでは実装しない）
- 集合場所・目的地の削除ボタンは、参照チェックの結果に応じて
  活性・非活性を切り替える
- 「コーチ名」欄は空欄可。入力時は`upsertCoachForFamily`を呼び、
  当該家庭に紐づく`Coach`を作成・更新する。家庭編集時は既存の
  コーチ名（`getCoachByFamilyId`の結果）を初期値として表示する
- コーチ単体の「登録済み一覧」セクションは設けない
  （家庭一覧の家庭名の横にコーチ名を併記する程度でよい）


# 実装ルール

- TypeScriptで実装する
- anyは使用しない
- 関数コンポーネント・Hooksで実装する
- React Hooksのルールを守る
- MVPなので過剰な抽象化はしない
- Firestoreアクセスは`src/firestore/`配下の関数経由に統一し、
  コンポーネントから直接`firebase/firestore`を呼び出さない
