# Task007 配車画面

Version: 1.0

---

# 目的

05_画面設計.md 8章の配車画面（メイン）を実装する。

自動配車は簡易スタブロジック（集合場所順に定員まで詰めるだけ）で実装し、
本格的なアルゴリズムはTask009で差し替える前提とする。

配車結果（Carpool）をFirestoreへ保存する。

---

# 背景

Task007aで、イベント・回答（Family・Child・Response）は
Firestoreへ移行済みという前提で本Taskを進める（未着手の場合は
Task007aを先に完了させること）。

07_配車アルゴリズム.md（設計ドキュメント）・009_配車アルゴリズム.md
（実装Task）は現時点で未着手のため、本Taskでは画面を成立させるための
最小限のスタブロジックのみを実装する。スタブの判定結果（誰がどの車に
乗るか）は、Task009で本物のロジックに差し替わった際に変わってよい
（画面側のインターフェースが変わらなければよい）。

ドラッグ＆ドロップによる車間移動・Undo（戻す）はTask008のスコープ。
「+ 車を追加」ボタンのみ、本Taskで実装する（応援号の追加に必要なため）。

# 前提とする決定事項

- Carpoolの保存は05_画面設計.md 11章の方針通り、変更の都度Firestoreへ
  自動保存する（明示的な保存ボタンはない）
- AIの判断理由（8.3）は、スタブロジックで計算可能な
  「集合場所一致」「定員に余裕あり」「兄弟を同乗」「ドライバーの子」の
  4種のみ表示する。「経路上にある」はTask009で本物のロジックが
  入ってから追加する
- コーチの参加要否（`coachParticipating`）は02_要件定義.md 10章の
  必須条件としてすでに確定しているため、スタブロジックでも
  「車を出すコーチは除外」「車を出さないコーチは家庭の子供と同じ
  集合場所から乗客に含める」までは実装する
- ルーティングは `/events/:eventId/dispatch` とする
  （Task004の「自動配車へ→」ボタンの遷移先）
- 「+ 車を追加」で作成した車は、メンバーが空の状態で作成される
  （実際に人を乗せる操作はTask008のドラッグ機能が前提となるため、
  本Task単体では空の車を作れるところまでとする）

# 完了条件

- `/events/:eventId/dispatch` にアクセスすると配車画面が表示される
- 初回表示時（配車未作成）は、05_画面設計.md 8.1の通り未配車一覧と
  「配車作成」ボタンのみが表示される
- 「配車作成」ボタンを押すと、スタブロジックによる配車結果が生成され、
  Firestoreの`events/{eventId}/carpools`へ保存される
- 一覧表示（8.2.1相当、横スクロール、ドラッグ操作なし）で、
  各車の「車名」「乗車率」「巡回順」「乗車メンバー（色＋名前）」が
  表示される
- 車カードをタップすると詳細ビュー（8.2.2）が開き、乗車メンバーが
  1人ずつ表示される
- 乗車メンバーをタップするとAIの判断理由（8.3、前提4種）が表示される
- 集合場所ごとの色分け（8.4）が適用される
- 定員超過の車は枠が赤くなり警告が表示される
- 未配車の子供・コーチが残っている場合、未配車リストに表示される
- 「配車作成」ボタンは再実行可能で、実行前に確認ダイアログ
  （「現在の配車は失われます。作り直しますか？」）を表示する
- 「+ 車を追加」で、ドライバー未定または任意のドライバー名を指定した
  空の車（応援号を含む）を追加できる
- npm run build が成功する

# 実装内容

## 1. 型定義

`src/types.ts` に以下を追加する（04_データ設計.md 10章準拠）。

```typescript
export type CarpoolType = "通常" | "応援号";

export interface CarpoolMember {
  type: "child" | "coach";
  id: string;
}

export interface Carpool {
  id: string;
  driverFamilyId: string | null; // typeが応援号、またはドライバー未定の場合はnull
  driverName: string;
  carName: string | null;
  capacity: number;
  type: CarpoolType;
  pickupLocationIds: string[];
  routeOrder: string[]; // 巡回順（pickupLocationIdの並び）
  members: CarpoolMember[];
}
```

## 3. スタブ自動配車ロジック

`src/dispatch/stubDispatch.ts` を新規作成する。

入力：対象イベントの参加者（`status: "回答済み"`かつ子供ごとの
`isParticipating: true`の子供、および`coachParticipating: true`の
家庭紐づきコーチ）、家庭の車出し情報、集合場所マスタ。

アルゴリズム（簡易版）

1. 「未回答」の家庭は対象外とする（04_データ設計.md 9章）
2. `isParticipating: false`・`noOutwardRide: true`の子供は
   行きの配車対象から除外する（帰りも同様に`noReturnRide`で判定する。
   本Taskでは往路のみを対象に実装し、復路は往路と同じロジックを
   別途適用する形でよい）
3. `driverOutward: true`の家庭ごとに1台のCarpool（type: "通常"、
   driverFamilyId: 当該家庭のid、capacity: `capacityToday`または
   `Family.vehicleCapacity`）を作成する
4. 参加する子供を、集合場所（`pickupLocationId`）ごとにグルーピングする
5. 集合場所グループを順に、まだ空きのあるCarpoolへ定員まで詰めていく
   （経路・方向の最適化は行わない、単純に先頭から詰めるだけでよい）
6. ドライバーの子供は、必ずそのドライバー自身のCarpoolへ割り当てる
   （4・5より優先する）
7. 兄弟（同一familyIdの子供）は同じCarpoolへ割り当てる
   （割り当て済みの兄弟がいる場合はそのCarpoolを優先する）
8. `coachParticipating: true`の家庭紐づきコーチについて
   - 当該家庭の`driverOutward`がtrueの場合：Carpoolのmembersには
     含めない（運転者として扱う）
   - falseの場合：所属家庭の子供と同じ集合場所グループとして扱い、
     5と同様に定員まで詰める（子供が未登録の家庭は対象外とし、
     未配車リストにも出さない）
9. 定員を超えて詰め切れなかった参加者は`未配車`として結果に含める
   （エラーにはしない、05_画面設計.md 8.5参照）

出力：`Carpool[]`（未保存）と、割り当てられなかった参加者のリスト。

呼び出し側（DispatchPageコンポーネント）が、結果を
`replaceCarpools`でFirestoreへ保存する。

## 5. DispatchPage の実装

`src/pages/DispatchPage.tsx` を新規作成する。

05_画面設計.md 8章のワイヤーフレームに準拠し、以下の状態を持つ。

- 初回表示（8.1）：Carpoolが1件も無い場合、未配車一覧＋
  「配車作成」ボタンのみを表示する
- 配車済み表示（8.2.1）：Carpoolが1件以上ある場合、横スクロールで
  各車のカードを一覧表示する（未配車カードは常に一番左、
  空になれば自動的に非表示）
- 詳細ビュー（8.2.2）：車カードをタップすると開く。ドラッグ操作は
  無いため、タップ操作のみ実装する
- AIの判断理由（8.3）：メンバーをタップするとその場で開閉する
- 色分け（8.4）：集合場所ごとに色を割り当てる（同一イベント内で
  一貫していればよく、色の割り当てロジックは簡易的なもので可）
- 「配車作成」ボタン（8.5）：押下時に確認ダイアログを表示し、
  「はい」の場合のみスタブロジックを実行してFirestoreへ保存する

## 6. 「+ 車を追加」

一覧表示の一番右に、通常の車カードと同じ幅の「＋」カードを配置する。

タップすると簡易フォームを表示する。

```
車を追加
────────────────
種別 [通常][応援号]

（種別=通常の場合）
ドライバー ▼(登録済み家庭)

（種別=応援号の場合）
ドライバー名 [__________]（自由入力）

定員 [__]

      [追加]
```

- 種別「通常」の場合、`driverFamilyId`は選択した家庭のidとし、
  `driverName`はその家庭の`parentName`を用いる
- 種別「応援号」の場合、`driverFamilyId`はnull、`driverName`は
  自由入力の値を用いる
- 追加すると`addCarpool`が呼ばれ、Firestoreへ即時反映される
- 追加された車のmembersは空配列とする（3章参照）
