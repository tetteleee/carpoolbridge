# Task007 配車画面

Version: 1.1

---

# 目的

05_画面設計.md 8章の配車画面（メイン）を実装する。

自動配車は簡易スタブロジック（集合場所順に定員まで詰めるだけ）で実装し、
本格的なアルゴリズムはTask009で差し替える前提とする。

配車結果（Carpool）をFirestoreへ保存する。

---

# 背景

Task007aで、イベント・回答（Family・Child・Response）は
Firestoreへ移行済みという前提で本Taskを進める（未着手の場合は
Task007aを先に完了させること）。

07_配車アルゴリズム.md（設計ドキュメント）・009_配車アルゴリズム.md
（実装Task）は現時点で未着手のため、本Taskでは画面を成立させるための
最小限のスタブロジックのみを実装する。スタブの判定結果（誰がどの車に
乗るか）は、Task009で本物のロジックに差し替わった際に変わってよい
（画面側のインターフェースが変わらなければよい）。

ドラッグ＆ドロップによる車間移動・Undo（戻す）はTask008のスコープ。
「+ 車を追加」ボタンのみ、本Taskで実装する（応援号の追加に必要なため）。

# 前提とする決定事項

- Carpoolの保存は05_画面設計.md 11章の方針通り、変更の都度Firestoreへ
  自動保存する（明示的な保存ボタンはない）
- AIの判断理由（8.3）は、スタブロジックで計算可能な
  「集合場所一致」「定員に余裕あり」「兄弟を同乗」「ドライバーの子」の
  4種のみ表示する。「経路上にある」はTask009で本物のロジックが
  入ってから追加する。運転者（`isDriver: true`のメンバー）にはAI判断理由を
  表示しない（マッチングの結果ではなく、回答に基づき自動生成される
  ため）
- 定員は「運転者本人を含む総定員」として扱う（04_データ設計.md 3・9・
  10章準拠）。運転者（保護者・コーチ問わず）は必ずCarpool.membersに
  `isDriver: true`として含めるため、定員判定は常に
  `members.length > capacity`のみで行う（特別な+1調整は不要）
- コーチの参加要否（`coachParticipating`）は02_要件定義.md 10章の
  必須条件としてすでに確定しているため、スタブロジックでも
  「車を出すコーチは運転者としてmembersに含める」「車を出さない
  コーチは家庭の子供と同じ集合場所から乗客に含める」までは実装する
- いつもは参加しない保護者等（`additionalParentNames`、Task004a）は、
  所属家庭の子供と同じ集合場所から乗客として自動配車の対象に含める
- ルーティングは `/events/:eventId/dispatch` とする
  （Task004の「自動配車へ→」ボタンの遷移先）
- 「+ 車を追加」で作成した車は、選択した家庭を運転者
  （`type: "parent"`、`isDriver: true`）として自動的にmembersへ
  含めた状態で作成される（メンバーは空ではなく、運転者1名のみの
  状態で作成される。それ以外の人を乗せる操作はTask008のドラッグ機能が
  前提となる）

# 完了条件

- `/events/:eventId/dispatch` にアクセスすると配車画面が表示される
- 初回表示時（配車未作成）は、05_画面設計.md 8.1の通り未配車一覧と
  「配車作成」ボタンのみが表示される
- 「配車作成」ボタンを押すと、スタブロジックによる配車結果が生成され、
  Firestoreの`events/{eventId}/carpools`へ保存される
- 一覧表示（8.2.1相当、横スクロール、ドラッグ操作なし）で、
  各車の「車名」「乗車率」「巡回順」「乗車メンバー（色＋名前）」が
  表示される
- 車カードをタップすると詳細ビュー（8.2.2）が開き、乗車メンバーが
  1人ずつ表示される
- 乗車メンバーをタップするとAIの判断理由（8.3、前提4種）が表示される
- 集合場所ごとの色分け（8.4）が適用される
- 定員超過の車は枠が赤くなり警告が表示される
- 未配車の子供・コーチが残っている場合、未配車リストに表示される
- 「配車作成」ボタンは再実行可能で、実行前に確認ダイアログ
  （「現在の配車は失われます。作り直しますか？」）を表示する
- 「+ 車を追加」で、ドライバー未定または任意のドライバー名を指定した
  空の車（応援号を含む）を追加できる
- npm run build が成功する

# 実装内容

## 1. 型定義

`src/types.ts` に以下を追加する（04_データ設計.md 10章準拠）。

```typescript
export type CarpoolType = "通常" | "応援号";

export type CarpoolMember =
  | { type: "child"; id: string }
  | { type: "coach"; id: string; isDriver: boolean }
  | { type: "parent"; familyId: string; name: string; isDriver: boolean }
  | { type: "other"; name: string };

export interface Carpool {
  id: string;
  driverFamilyId: string | null; // typeが応援号、またはドライバー未定の場合はnull
  driverName: string;
  carName: string | null;
  capacity: number; // 運転者本人を含む総定員
  type: CarpoolType;
  pickupLocationIds: string[];
  routeOrder: string[]; // 巡回順（pickupLocationIdの並び）
  members: CarpoolMember[]; // 運転者も必ずisDriver:trueのメンバーとして含む
}
```

`type: "parent"`は個人単位のマスタを持たないため、`id`ではなく
`familyId`＋`name`（`Family.parentName`または`additionalParentNames`の
自由入力）で表現する（04_データ設計.md 10章参照）。

## 3. スタブ自動配車ロジック

`src/dispatch/stubDispatch.ts` を新規作成する。

入力：対象イベントの参加者（`status: "回答済み"`かつ子供ごとの
`isParticipating: true`の子供、`coachParticipating: true`の
家庭紐づきコーチ、および`additionalParentNames`）、家庭の車出し情報、
集合場所マスタ。

アルゴリズム（簡易版）

1. 「未回答」の家庭は対象外とする（04_データ設計.md 9章）
2. `isParticipating: false`・`noOutwardRide: true`の子供は
   行きの配車対象から除外する（帰りも同様に`noReturnRide`で判定する。
   本Taskでは往路のみを対象に実装し、復路は往路と同じロジックを
   別途適用する形でよい）
3. `driverOutward: true`の家庭ごとに1台のCarpool（type: "通常"、
   driverFamilyId: 当該家庭のid、capacity: `capacityToday`または
   `Family.vehicleCapacity`、運転者本人を含む総定員）を作成する。
   運転者を表すメンバーを、作成と同時にmembersへ1件追加する
   （4章参照、コーチが運転する場合はcoach、それ以外はparent）
4. 参加する子供を、集合場所（`pickupLocationId`）ごとにグルーピングする
5. 集合場所グループを順に、まだ空きのあるCarpoolへ定員まで詰めていく
   （経路・方向の最適化は行わない、単純に先頭から詰めるだけでよい。
   運転者はすでにmembersに1名分含まれているため、定員のうち
   残り`capacity - 1`人分が子供・その他乗客の枠となる）
6. ドライバーの子供は、必ずそのドライバー自身のCarpoolへ割り当てる
   （4・5より優先する）
7. 兄弟（同一familyIdの子供）は同じCarpoolへ割り当てる
   （割り当て済みの兄弟がいる場合はそのCarpoolを優先する）
8. `coachParticipating: true`の家庭紐づきコーチについて
   - 当該家庭の`driverOutward`がtrueの場合：3章で作成した運転者
     メンバーを`{ type: "coach", id, isDriver: true }`とする
     （`type: "parent"`の運転者は生成しない。1家庭1台のため
     運転者は1人）
   - falseの場合：`{ type: "coach", id, isDriver: false }`として、
     所属家庭の子供と同じ集合場所グループで扱い、5と同様に
     定員まで詰める（子供が未登録の家庭は対象外とし、
     未配車リストにも出さない）
9. `additionalParentNames`（家庭単位で複数の場合あり）について、
   各名前を`{ type: "parent", familyId, name, isDriver: false }`として、
   所属家庭の子供と同じ集合場所グループで扱い、5と同様に定員まで
   詰める（子供が未登録の家庭は対象外とし、未配車リストにも出さない）
10. 定員を超えて詰め切れなかった参加者は`未配車`として結果に含める
    （エラーにはしない、05_画面設計.md 8.5参照）

出力：`Carpool[]`（未保存）と、割り当てられなかった参加者のリスト。

呼び出し側（DispatchPageコンポーネント）が、結果を
`replaceCarpools`でFirestoreへ保存する。

## 4. AIの判断理由の算出

`src/dispatch/reasons.ts` を新規作成する。

```typescript
export type AssignmentReason =
  | "集合場所一致"
  | "定員に余裕あり"
  | "兄弟を同乗"
  | "ドライバーの子";

export function getAssignmentReasons(
  member: CarpoolMember,
  carpool: Carpool,
  context: DispatchContext // 家庭・子供・集合場所等の参照データ
): AssignmentReason[] { ... }
```

該当する理由のみを配列で返す（05_画面設計.md 8.3参照）。
「経路上にある」は本Taskでは算出しない（前提4参照）。
`isDriver: true`のメンバー（運転者）には理由を算出・表示しない
（回答に基づき自動生成されたメンバーであり、マッチングの結果では
ないため）。`type: "other"`のメンバーも同様に対象外とする。

## 5. DispatchPage の実装

`src/pages/DispatchPage.tsx` を新規作成する。

05_画面設計.md 8章のワイヤーフレームに準拠し、以下の状態を持つ。

- 初回表示（8.1）：Carpoolが1件も無い場合、未配車一覧＋
  「配車作成」ボタンのみを表示する
- 配車済み表示（8.2.1）：Carpoolが1件以上ある場合、横スクロールで
  各車のカードを一覧表示する（未配車カードは常に一番左、
  空になれば自動的に非表示）
- 詳細ビュー（8.2.2）：車カードをタップすると開く。ドラッグ操作は
  無いため、タップ操作のみ実装する
- AIの判断理由（8.3）：メンバーをタップするとその場で開閉する
- 色分け（8.4）：集合場所ごとに色を割り当てる（同一イベント内で
  一貫していればよく、色の割り当てロジックは簡易的なもので可）
- 「配車作成」ボタン（8.5）：押下時に確認ダイアログを表示し、
  「はい」の場合のみスタブロジックを実行してFirestoreへ保存する

## 6. 「+ 車を追加」

一覧表示の一番右に、通常の車カードと同じ幅の「＋」カードを配置する。

タップすると簡易フォームを表示する。

```
車を追加
────────────────
種別 [通常][応援号]

（種別=通常の場合）
ドライバー ▼(登録済み家庭)

（種別=応援号の場合）
ドライバー名 [__________]（自由入力）

定員 [__]

      [追加]
```

- 種別「通常」の場合、`driverFamilyId`は選択した家庭のidとし、
  `driverName`はその家庭の`parentName`を用いる。あわせて
  `{ type: "parent", familyId, name: parentName, isDriver: true }`を
  membersへ1件自動追加する
- 種別「応援号」の場合、`driverFamilyId`はnull、`driverName`は
  自由入力の値を用いる。この場合はmembersへ運転者メンバーを
  追加しない（応援号の運転者はマスタに紐づかないため、
  `driverName`表示のみで表す。既存のCarpool.driverName運用を踏襲）
- 追加すると`addCarpool`が呼ばれ、Firestoreへ即時反映される
- 追加された車のmembersは、種別「通常」の場合は運転者1名のみ、
  種別「応援号」の場合は空配列とする
