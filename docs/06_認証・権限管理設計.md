# 06_認証・権限管理設計

Version: 1.1 (MVP)

---

# 1. 目的

本システムは学童野球チーム内のみで利用する配車支援システムである。

Firebase Hostingでインターネットへ公開されるため、
URLを知っている第三者から子供・保護者情報へアクセスできないよう、
最小限のアクセス制御を実装する。

MVPでは

- セキュリティ
- 実装容易性
- 運用容易性

のバランスを重視する。

---

# 2. 設計方針

採用方式

- Firebase Authentication（Anonymous Authentication）
- UIDホワイトリスト方式

利用者は数名（管理者・配車担当）のみであるため、

- メール認証
- パスワード認証
- 招待コード
- 利用申請画面
- 承認画面

は実装しない。

初回アクセス時に表示されるUIDを利用者が管理者へLINE等で送信し、
管理者がFirebase Consoleから `staffUsers` コレクションへ登録する。

利用者が増えた場合は

- pendingUsers
- 承認画面
- ロール管理

へ拡張可能な設計とする。

# 6. Firestore Security Rules

MVPでは `staffUsers` に登録されたUIDのみ、
アプリデータへのアクセスを許可する。

```javascript
rules_version = '2';

service cloud.firestore {
  match /databases/{database}/documents {

    function isStaff() {
      return request.auth != null &&
        exists(
          /databases/$(database)/documents/staffUsers/$(request.auth.uid)
        );
    }

    // 家庭
    match /families/{document=**} {
      allow read, write: if isStaff();
    }

    // 子供
    match /children/{document=**} {
      allow read, write: if isStaff();
    }

    // コーチ
    match /coaches/{document=**} {
      allow read, write: if isStaff();
    }

    // 集合場所
    match /pickupLocations/{document=**} {
      allow read, write: if isStaff();
    }

    // 目的地
    match /destinations/{document=**} {
      allow read, write: if isStaff();
    }

    // イベント
    match /events/{document=**} {
      allow read, write: if isStaff();
    }

    // staffUsers
    match /staffUsers/{uid} {

      allow read:
        if request.auth != null &&
           request.auth.uid == uid;

      allow write:
        if false;
    }

    // その他は拒否
    match /{document=**} {
      allow read, write: if false;
    }
  }
}
```

# 7. 権限

MVPではFirestore Rulesによる権限制御は

**staffUsersへ登録されているかどうか**

のみ判定する。

そのため、

- 管理者
- 配車担当

はFirestore Rules上では同一権限となる。

以下の役割分担は運用上の区別であり、
Firestore Rulesでは区別しない。

|操作|管理者|配車担当|
|---|---|---|
|イベント作成・編集|○|○|
|回答入力|○|○|
|配車作成|○|○|
|LINE共有|○|○|
|家庭マスタ編集|○|○|
|子供マスタ編集|○|○|
|集合場所編集|○|○|
|目的地編集|○|○|

将来的には

```
role

admin

dispatcher
```

などのロール管理へ拡張可能とする。

# 9. 非採用案

## メール認証

採用しない。

理由

- アカウント作成が必要
- MVPには過剰

---

## パスワード認証

採用しない。

理由

- パスワード管理が必要
- 毎年引継ぎが発生する

---

## 招待コード

採用しない。

理由

- 漏洩リスク
- 更新運用が必要
- MVPには過剰

---

## pendingUsers方式

今回は採用しない。

理由

- 利用者が数名しかいない
- 承認画面を作るメリットが小さい
- Firebase Consoleで登録した方が運用が簡単

利用者が増えた場合は導入を検討する。

---

# 10. 採用理由

MVPでは

- 実装が簡単
- セキュリティを確保できる
- 運用コストが極めて低い

ことを重視した。

利用者は数名であるため、

「初回だけUIDをLINEで送る」

という運用は十分許容できる。

Firebase Authenticationの匿名認証を利用することで、
パスワード管理を不要としつつ、
UIDホワイトリスト方式のみで安全なアクセス制御を実現できる。

将来的に利用者が増えた場合は

- pendingUsers
- 承認画面
- ロール管理

を追加することで、既存設計を大きく変更することなく拡張できる。
