# 05_画面設計

Version: 1.0

# 1. 目的

配車担当・管理者が実際に使う画面を、実装可能なレベルまで定義する。

本ドキュメントは元々「プロトタイプ」として、配車担当に見せて検証する
想定で作成したが、今回は開発者（Kenshiro）自身が確認・判断する運用と
したため、検証用ドキュメントと実装用仕様書を分けず、本ドキュメント1本に
統合した。

---

# 2. アプリ名

**配車アシスタント**

---

# 3. 設計方針（配車担当の思考順に合わせる）

配車担当は概ね以下の順番で考える（01_現状業務分析・07_配車アルゴリズムと共通）。

```
集合場所ごとに分ける
↓
車を決める
↓
定員確認
↓
方向確認
↓
微調整
```

この思考順に沿って、画面上でも
「集合場所が色でひと目に分かる」→「車ごとにまとまって見える」→「定員・方向がすぐ確認できる」
という情報の出し方を優先する。

このアプリの最重要価値は、

> 「配車作成」ボタンを押して、80〜90%完成した配車案が10秒程度で出てくること

である。配車担当が一番困っているのは「定員オーバー」と「乗せ忘れ」なので、
この2つは常に画面上で目立たせる。

# 7. イベント編集（配車担当がLINEを見ながら入力）

参加・帰り・車出し・乗車可能人数・特記事項など、イベントに関する回答情報を
まとめて編集する画面。単なる「回答の入力」ではなく、後からいつでも
修正できる「編集」画面として位置づける。

家庭ごとにカードをまとめ、その中に所属する子供（1人〜複数人）の回答を並べる。
LINE返信自体が家庭単位で届くため、入力の単位も家庭に合わせる。

```
┌─────────────────────────┐
│ 7/12(土) 練習試合          │
├─────────────────────────┤
│ 山田家                    │
│  状態 [未回答 / 回答済]   │
│                          │
│  車出し(行き) [可][不可]  │
│  車出し(帰り) [可][不可]  │
│  乗車可能人数 [ 5 ]人      │ ← 未変更(null)時は通常定員を薄字表示
│  ─────────────────      │
│  山田 太郎（小6）          │
│   参加 [○][✕]              │
│   □ 行きの配車不要         │
│   □ 帰りの配車不要         │
│  備考 [______________]  │
│  ─────────────────      │
│  山田 花（小3）            │
│   参加 [○][✕]              │
│   □ 行きの配車不要         │
│   □ 帰りの配車不要         │
│  備考 [______________]  │
├─────────────────────────┤
│ 佐藤家                    │
│  状態 [未回答 / 回答済]   │
│                          │
│  車出し(行き) [可][不可]  │
│  車出し(帰り) [可][不可]  │
│  乗車可能人数 [ 4 ]人 ⚠️   │ ← 変更時は数値を保存し、アイコン等で強調
│  ─────────────────      │
│  佐藤 花子（小4）          │
│   参加 [○][✕]              │
│   □ 行きの配車不要         │
│   □ 帰りの配車不要         │
│   備考 [______________]  │
│  ...                    │
├─────────────────────────┤
│ ← イベント一覧　配車画面へ →│
└─────────────────────────┘
```

- 乗車可能人数は、未変更時（`capacityToday: null`）は `Family.vehicleCapacity` を薄字でプレースホルダー的に表示する。
  当日の荷物等で人数が変動する場合は、担当者が数値を上書き変更できる。変更されると `capacityToday` に数値が保存され、画面上では通常字＋アイコン等で「変更済み」であることが分かるように表示する
- 家庭に紐づくコーチ（04_データ設計.md 5章の`familyId`設定済みコーチ）が
  存在する場合のみ、家庭カード内に以下を追加表示する（該当コーチがいない
  家庭では非表示）。

  ```
  ─────────────────
  コーチ（〇〇） 参加 [○][✕]
  ```

  車を出すかどうかの入力欄は別途設けない。家庭自体の車出し（行き/帰り）が
  「可」であれば、参加中のコーチがその運転者として扱われ乗車人数には
  カウントしない。「不可」の場合は所属家庭の子供と同じ集合場所から
  乗客として自動配車の対象に含める（02_要件定義.md 8・10章参照）。
- 家庭カードの上部に「状態（未回答 / 回答済）」のトグルを置く。初期状態は「未回答」。
  配車担当がいずれかの項目（参加・車出し等）を操作して変更すると、自動的に「回答済」に切り替わる（手動で戻すことも可能）。
- カード内に子供ごとの回答（参加・行きの配車不要・帰りの配車不要・備考）を
  1人ずつ並べる（兄弟が複数人いる家庭でも、1枚のカードで完結する）
- 「現地集合」「保護者迎え」などの特殊なケースもすべて、「行きの配車不要」「帰りの配車不要」のチェックボックスで表現する。それ以外の例外事項は備考欄に入力する
- 家庭カードは並び替えできるようにする。並び替え基準は以下の4種類を用意する
  - 五十音順
  - 集合場所順
  - 学年順
  - 未回答優先（未回答の家庭を上に表示）
  画面上部にドロップダウンまたはタブで並び替え基準を切り替えられるようにする
- 「保存」ボタンは置かない。入力した瞬間に自動保存する（11章の自動保存方針と統一）。
  画面下部には「← イベント一覧」「配車画面へ →」のナビゲーションのみ配置する

# 8. 配車画面（メイン）

## 8.1 初回表示（自動配車前）

```
┌─────────────────────────┐
│ 配車結果                  │
├─────────────────────────┤
│                         │
│      [配車作成]          │
│                         │
├─────────────────────────┤
│ 未配車 18名              │
│                         │
│ 🟦 山田太郎 (小4)       │
│ 🟨 佐藤花子 (小6) ⚠️未回答      │
│ 🟩 田中一郎 (小5)              │
└─────────────────────────┘
```

- 起動直後は未配車エリアのみ表示する
- 「配車作成」ボタンを押すと自動配車を実行する
- 子供カードの色は集合場所ごとに色分けする（8.3参照）
- 未回答の家庭に属する子どもには「⚠️未回答」と目立つように表示する

---

## 8.2 自動配車後

画面全体を**縦スクロール**とする。

画面は以下の順に表示する。

```
操作ボタン
    ↓
未配車
    ↓
車カード
    ↓
車カード
    ↓
車カード
```

一覧画面・詳細画面は分けず、1画面ですべての配車状況を確認・編集できる構成とする。

---

### 8.2.1 操作エリア

```
┌─────────────────────────┐
│ 配車結果                  │
├─────────────────────────┤
│ [人を追加] [車を追加]     │
│                         │
│ [移動を戻す]     [共有]   │
└─────────────────────────┘
```

表示する操作は以下とする。

- 人を追加
- 車を追加（通常車・応援号）
- 移動を戻す（Undo）
- LINE共有

---

### 8.2.2 未配車エリア

```
┌─────────────────────────┐
│ 未配車　3名              │
├─────────────────────────┤
│ ≡ 子ども 📍西公園 山田太郎 (小4)       │
│ ≡ 保護者 📍西公園 中村恵子        │
│ ≡ 子ども 📍中央公園 石川蒼 (小3)          │
└─────────────────────────┘
```

- 未配車エリアは常に画面最上部へ表示する
- 未配車が0人になった場合は自動的に非表示とする
- 人カードはドラッグ＆ドロップ可能とする
- 未回答の家庭は⚠️未回答を表示する

---

### 8.2.3 車カード

```
┌─────────────────────────┐
│ 🚗 栗本コーチ            4/5 │
│ コーチ・ドライバー       │
│                         │
│ 📍西公園                │
│ 📍中央公園              │
├─────────────────────────┤
│ ≡ 子ども　📍西公園 山田太郎       │
│ ≡ 子ども　📍中央公園 佐藤花子       │
│ ≡ 保護者　中村恵子       │
└─────────────────────────┘
```

車カードには以下を表示する。

- 車アイコン
- 車名
- ドライバー種別（保護者・コーチ）
- 乗車率（4/5など）
- 巡回する集合場所
- 乗車メンバー

定員超過の場合はカード枠を赤くし、⚠マークを表示する。

---

### 8.2.4 人カード

人カードはドラッグ可能なカードとして表示する。

```
≡

子ども

山田太郎
```

または

```
≡

保護者

中村恵子
```

または

```
≡

コーチ

田中コーチ
```

人カードには以下を表示する。

- ドラッグハンドル
- 種別ラベル（子ども・保護者・コーチ）
- 集合場所ラベル
- 氏名

子供カードの色は集合場所ごとに色分けする（8.3参照）。

---

### 8.2.5 ドラッグ＆ドロップ

人カードは

- 未配車エリア
- 各車カード

の間を自由にドラッグ＆ドロップできる。

ドラッグ可能であることが分かるよう、カード左側にドラッグハンドル（≡）を表示する。

ドラッグ中はドロップ可能な車カードを強調表示し、配置先が分かりやすいUIとする。

ドラッグは350ms以上の長押しで開始する。タップやスクロールとドラッグを誤操作なく使い分けられるようにする。

---

## 8.3 色分けルール

- 集合場所ごとに色を割り当て、子供カードの左側やラベルに反映する
- 保護者、子供、コーチのカードは共通デザインとし、種別ラベルのみで区別する
- 集合場所を示すラベルも人のカードに表示する（例：📍西公園）
- 未回答の家庭に属する子どもには「⚠️未回答」を表示する

---

## 8.4 配車作成（自動配車）ボタン

- ボタン名は「配車作成」
- 実行するたびに以下の確認ダイアログを表示する

```
現在の配車は失われます。
作り直しますか？

     [はい]   [いいえ]
```

- 何度でも再実行可能
- 定員に対して配車しきれない人数がいる場合は、配車可能なところまで実施し、残りは「未配車」として表示する（エラーにはしない）

---


# 10. 管理（家庭・子供・集合場所登録）

UC-09〜UC-11に対応する、マスタデータ登録用の画面。
Milestone2で暫定実装済み（見た目の作り込みは行っておらず、
入力・保存できることのみを優先した最小限のフォーム）。

```
┌─────────────────────────┐
│ 管理（暫定）               │
├─────────────────────────┤
│ 集合場所を登録              │
│  名称     [__________]   │
│  住所     [__________]   │
│  緯度     [______]       │
│  経度     [______]       │
│         [登録]           │
├─────────────────────────┤
│ 家庭を登録                 │
│  家庭名   [__________]   │
│  保護者名 [__________]   │
│  通常乗車人数 [__]        │
│         [登録]           │
├─────────────────────────┤
│ 子供を登録                 │
│  名前     [__________]   │
│  学年     ▼小1〜小6       │
│  所属家庭 ▼(登録済み家庭)  │
│  集合場所 ▼(登録済み場所)  │
│         [登録]           │
├─────────────────────────┤
│ 登録済み一覧                │
│  集合場所:西公園, 中央公園… │
│  家庭:山田家（山田太郎）定員5 │
│  子供:山田太郎（小6）…      │
└─────────────────────────┘
```

- 集合場所→家庭→子供の順で登録する（子供登録時に家庭・集合場所の
  選択肢が必要なため、この順序を画面上でも維持する）
- 緯度経度は現時点では手入力（Google Maps検索連携はPhase3、
  6章の目的地登録と同様の制約）
- この画面は「使いやすさ」より「登録できること」を優先した暫定UIであり、
  デザイン・入力補助（住所からの緯度経度自動取得など）は今回のMVP後半
  またはPhase2で見直す

# 11. 保存・全般仕様

- 修正内容は最後に変更した状態を自動保存する（明示的な保存ボタンは不要）
- 配車品質スコアのような数値表示は行わない（担当者の総合判断を尊重する）

# 12. 確定した仕様（旧プロトタイプ版で未確定だった4項目）

前バージョン（プロトタイプ）で保留にしていた4項目は、開発者本人が
確認・判断する運用に変更したことに伴い、以下の通り確定した。

## 画面下部固定ボタン

**採用しない**。ホーム画面・イベント編集画面・配車画面など、
どの画面も各画面固有のボタン（配車作成・戻す・+イベント作成等）だけで
操作が完結するため、共通の下部固定ナビゲーションは置かない。

## ダークモード

**MVPでは実装しない**。将来的に必要になった時点で08_MVP開発計画
以降のフェーズで検討する。

## 「+ 車を追加」ボタン

一覧（8.2.1）の一番右に、通常の車カードと同じ幅の「＋」カードとして配置する。

## 未配車カードの表示位置

常に一番左に固定する。未配車の子供がいなくなれば自動的に非表示になる。

# 13. 検証・確認したいポイント（開発者自身によるセルフチェック項目）

配車担当への外部ヒアリングは行わない方針としたため、以下は開発者自身が
実装後に一通り操作して確認する項目とする。

- 「配車作成」ボタンを押してから配車案が出るまでの流れは分かりやすいか
- 定員超過・未配車の表示は目に留まりやすいか
- ドラッグ&ドロップでの修正は直感的か（2タップ方式の方が良いか）
- AIの判断理由（集合場所一致・経路上にある・定員に余裕あり・兄弟を同乗など）は
  納得感があるか、情報が多すぎないか
- 「+ 車を追加」カードは一覧の右端にあっても自然に見つけられるか
- LINE共有テキストの見やすさは現行のLINE投稿と比べてどうか
- タップ（AI理由表示）と長押し+ドラッグ（車移動）のジェスチャーが
  誤操作なく使い分けられるか
