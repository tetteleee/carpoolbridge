# 05_画面設計

Version: 1.0

---

# 1. 目的

配車担当・管理者が実際に使う画面を、実装可能なレベルまで定義する。

本ドキュメントは元々「プロトタイプ」として、配車担当に見せて検証する
想定で作成したが、今回は開発者（Kenshiro）自身が確認・判断する運用と
したため、検証用ドキュメントと実装用仕様書を分けず、本ドキュメント1本に
統合した。

---

# 2. アプリ名

**配車アシスタント**

---

# 3. 設計方針（配車担当の思考順に合わせる）

配車担当は概ね以下の順番で考える（01_現状業務分析・07_配車アルゴリズムと共通）。

```
集合場所ごとに分ける
↓
車を決める
↓
定員確認
↓
方向確認
↓
微調整
```

この思考順に沿って、画面上でも
「集合場所が色でひと目に分かる」→「車ごとにまとまって見える」→「定員・方向がすぐ確認できる」
という情報の出し方を優先する。

このアプリの最重要価値は、

> 「配車作成」ボタンを押して、80〜90%完成した配車案が10秒程度で出てくること

である。配車担当が一番困っているのは「定員オーバー」と「乗せ忘れ」なので、
この2つは常に画面上で目立たせる。

# 7. イベント編集（配車担当がLINEを見ながら入力）

参加・帰り・車出し・乗車可能人数・特記事項など、イベントに関する回答情報を
まとめて編集する画面。単なる「回答の入力」ではなく、後からいつでも
修正できる「編集」画面として位置づける。

家庭ごとにカードをまとめ、その中に所属する子供（1人〜複数人）の回答を並べる。
LINE返信自体が家庭単位で届くため、入力の単位も家庭に合わせる。

```
┌─────────────────────────┐
│ 7/12(土) 練習試合          │
├─────────────────────────┤
│ 山田家                    │
│                          │
│  車出し(行き) [可][不可]  │
│  車出し(帰り) [可][不可]  │
│  乗車可能人数 [ 5 ]人      │ ← 通常乗車人数を初期表示、変更可
│  ─────────────────      │
│  山田 太郎（小6）          │
│   参加 [○][✕]              │
│   □ 行きの配車不要         │
│   □ 帰りの配車不要         │
│   備考 [______________]  │
│  ─────────────────      │
│  山田 花（小3）            │
│   参加 [○][✕]              │
│   □ 行きの配車不要         │
│   □ 帰りの配車不要         │
│   備考 [______________]  │
├─────────────────────────┤
│ 佐藤家                    │
│                          │
│  車出し(行き) [可][不可]  │
│  車出し(帰り) [可][不可]  │
│  乗車可能人数 [ 4 ]人      │ ← 通常乗車人数を初期表示、変更可
│  ─────────────────      │
│  佐藤 花子（小4）          │
│   参加 [○][✕]              │
│   □ 行きの配車不要         │
│   □ 帰りの配車不要         │
│   備考 [______________]  │
│  ...                    │
├─────────────────────────┤
│ ← イベント一覧　配車画面へ →│
└─────────────────────────┘
```

- 乗車可能人数は、Family.vehicleCapacity（通常乗車人数）を初期値として
  自動表示する。当日は荷物量等により人数が変動することがあるため、
  配車担当が画面上で自由に上書き変更できる（保存値は04_データ設計.mdの
  Response.capacityToday）
- 家庭カードの上部に、その家庭に共通する情報（車出し可否・乗車可能人数）をまとめる
- カード内に子供ごとの回答（参加・行きの配車不要・帰りの配車不要・備考）を
  1人ずつ並べる（兄弟が複数人いる家庭でも、1枚のカードで完結する）
- 「現地集合」「保護者迎え」などの特殊なケースもすべて、「行きの配車不要」「帰りの配車不要」のチェックボックスで表現する。それ以外の例外事項は備考欄に入力する
- 家庭カードは並び替えできるようにする。並び替え基準は以下の4種類を用意する
  - 五十音順
  - 集合場所順
  - 学年順
  - 未回答優先（未回答の家庭を上に表示）
  画面上部にドロップダウンまたはタブで並び替え基準を切り替えられるようにする
- 「保存」ボタンは置かない。入力した瞬間に自動保存する（11章の自動保存方針と統一）。
  画面下部には「← イベント一覧」「配車画面へ →」のナビゲーションのみ配置する

# 8. 配車画面（メイン）

## 8.1 初回表示（自動配車前）

```
┌─────────────────────────┐
│ 7/12(土) 配車             │
├─────────────────────────┤
│ 未配車: 18人               │
│                          │
│ 🟦 山田太郎 小6 西明石駅   │
│ 🟨 佐藤花子 小4 大久保駅   │
│ 🟩 田中一郎 小5 大久保駅   │
│ ...                      │
│                          │
│      [配車作成]           │
└─────────────────────────┘
```

- 起動直後は未配車の子供一覧のみを表示し、「配車作成」ボタンを押してもらう
- 子供カードの色は集合場所ごとに色分け（8.4参照）

## 8.2 自動配車後（メイン画面）

画面はスマホ幅に収めるため、「一覧（概要）」と「詳細（タップで展開）」の2段構成にする。

### 8.2.1 一覧（概要）— 横スクロールで全車を俯瞰

```
┌───────┐┌──────────┬──────────┬────────┐
│ ↺戻す  ││ 未配車     │ 栗本号     │ 大船号   │…│  ➕   │
├───────┘│ 2人       │ 4/5       │ 3/4     │  │ 車を  │
         │           │ 西明石     │ 大久保   │  │ 追加  │
         │           │  ↓        │  ↓      │  │       │
         │           │ 大久保     │ 会場     │  │       │
         │           │  ↓        │         │  │       │
         │           │ 会場       │         │  │       │
         ├──────────┼──────────┼─────────┤  │       │
         │🟩鈴木次郎  │🟦山田太郎  │🟨田中一郎 │  │       │
         │🟥高橋四郎  │🟦山田花    │🟨伊藤三郎 │  │       │
         │           │           │🟩佐藤花子 │  │       │
         └──────────┴──────────┴─────────┘  └───────┘
```

- 車ごとに「車名」「乗車率（4/5等）」「巡回順（経由地）」をカード上部にまとめて表示する。
  詳細ビューを開かなくても一覧だけで巡回順が分かるようにする
- その下に、乗車メンバーを「色＋名前」の
  簡易チップで並べる（学年・集合場所名などの詳細は省き、ドラッグ操作で
  識別できる最低限の情報にとどめる）
- 定員超過の場合はカード枠を赤くし、⚠マークを表示
- **未配車カードは常に一番左に固定する**。未配車の子供がいなくなれば
  自動的に非表示になる（カードごと消える）
- **「+ 車を追加」は一覧の一番右に、通常の車カードと同じ幅の「＋」カードとして配置する**。
  車出し回答の有無に関わらず車を追加できる（応援号もこのカードから追加する。
  特別な専用ボタンは用意しない）
- **「戻す」ボタンは画面左上（一覧の外・スクロールに影響されない位置）に常設する**。
  直前の修正を1手戻せる
- 子供チップをドラッグして別の車の枠へドロップすると、その場で車が入れ替わる
- 車カード（栗本号・大船号など）をタップすると8.2.2の詳細ビューが開く
- 操作の競合を避けるため、子供チップは「タップ＝AI判断理由を表示（8.3）」
  「長押し＋ドラッグ＝車の移動」とジェスチャーを使い分ける（要検証:
  実際に使ってみて違和感がないか確認する）

### 8.2.2 詳細（タップで展開）— 1台分をフル表示

```
┌─────────────────────────┐
│ 栗本号                    │
│ 4/5                      │
│                          │
│ 西公園                    │
│  ↓                       │
│ 中央公園                   │
│  ↓                       │
│ 会場                      │
├─────────────────────────┤
│ 🟦 山田 太郎               │
│    小6                   │
│    西公園                 │
├─────────────────────────┤
│ 🟦 山田 花                 │
│    小3                   │
│    西公園                 │
├─────────────────────────┤
│ 🟩 田中 一郎               │
│    小5                   │
│    中央公園                │
├─────────────────────────┤
│         [閉じる]          │
└─────────────────────────┘
```

- 車名・乗車率・経由地（巡回順）をカード上部にまとめて表示
- その下に、乗車する子供を1人ずつ区切り線で分けて縦に並べる
  （集合場所の色ドット・氏名・学年・集合場所名を1人分のブロックとして表示）
- タップした子供のブロックをさらにタップすると、8.3のAI判断理由がその場で開く
  （詳細ビューでは車間移動を行わないため、タップ操作のみでよい）
- ドラッグ&ドロップでの車間移動は、8.2.1の一覧表示（複数車が同時に見える状態）で行う。
  詳細ビュー（1台のみ表示）を開いている間は、他の車が画面上に無いため
  ドラッグ移動はできない仕様とする（この点は要検証）

## 8.3 子供カードの詳細・AIの判断理由

子供カードをタップすると、AIがその割り当てを選んだ理由を表示する。
その子に実際に適用された条件だけを表示する（該当しない項目は出さない）。

```
┌─────────────────────────┐
│ 山田 太郎                 │
│ 小6                      │
│ 西明石駅                  │
├─────────────────────────┤
│ 集合場所一致               │
│ 経路上にある               │
│ 定員に余裕あり             │
│ 兄弟を同乗                 │
└─────────────────────────┘
```

表示しうる理由の例（該当する場合のみ表示）

- 集合場所一致:通常の集合場所とその車の巡回経路が一致している
- 経路上にある:目的地までの方向・巡回順に沿っている（07_配車アルゴリズムの
  方位角クラスタリングの結果）
- 定員に余裕あり:その車の定員に対して詰め込みすぎていない
- 兄弟を同乗:兄弟が同じ車に割り当てられている（兄弟がいる子供のみ表示）
- ドライバーの子:その車のドライバーの子供である（該当する場合のみ表示）

複数の理由が同時に成立する場合はすべて列挙する。
07_配車アルゴリズムのどの判定ロジックが「理由」として文言化できるかは、
実装時に対応関係を精査する必要がある（デバッグ用途としても活用できるよう、
できるだけロジックの判定結果をそのまま理由として出す設計にする）。

## 8.4 色分けルール

- 集合場所ごとに色を割り当て、子供カードの名前の左に色付きの四角（🟦🟨🟩など）で表示する
- カード自体の背景色は変えない（車カードは全て白背景で統一し、情報の見やすさを優先）

## 8.5 配車作成（自動配車）ボタン

- ボタン名は「配車作成」
- 実行するたびに以下の確認ダイアログを表示する

```
現在の配車は失われます。
作り直しますか？

     [はい]   [いいえ]
```

- 何度でも再実行可能
- 定員に対して配車しきれない人数がいる場合は、配車可能なところまで実施し、
  残りは「未配車」として表示する（エラーにはしない）

UC-09〜UC-11に対応する、マスタデータ登録用の画面。
Milestone2で暫定実装済み（見た目の作り込みは行っておらず、
入力・保存できることのみを優先した最小限のフォーム）。

```

│ 管理（暫定）               │
├─────────────────────────┤
│ 集合場所を登録              │
│  名称     [__________]   │
│  住所     [__________]   │
│  緯度     [______]       │
│  経度     [______]       │
│         [登録]           │
├─────────────────────────┤
│ 家庭を登録                 │
│  家庭名   [__________]   │
│  保護者名 [__________]   │
│  通常乗車人数 [__]        │
│         [登録]           │
├─────────────────────────┤
│ 子供を登録                 │
│  名前     [__________]   │
│  学年     ▼小1〜小6       │
│  所属家庭 ▼(登録済み家庭)  │
│  集合場所 ▼(登録済み場所)  │
```

- 集合場所→家庭→子供の順で登録する（子供登録時に家庭・集合場所の

- 緯度経度は現時点では手入力（Google Maps検索連携はPhase3、
  6章の目的地登録と同様の制約）
- この画面は「使いやすさ」より「登録できること」を優先した暫定UIであり、
  デザイン・入力補助（住所からの緯度経度自動取得など）は今回のMVP後半
  またはPhase2で見直す

# 11. 保存・全般仕様

- 修正内容は最後に変更した状態を自動保存する（明示的な保存ボタンは不要）
- 配車品質スコアのような数値表示は行わない（担当者の総合判断を尊重する）

# 12. 確定した仕様（旧プロトタイプ版で未確定だった4項目）

前バージョン（プロトタイプ）で保留にしていた4項目は、開発者本人が
確認・判断する運用に変更したことに伴い、以下の通り確定した。

## 画面下部固定ボタン

**採用しない**。ホーム画面・イベント編集画面・配車画面など、
どの画面も各画面固有のボタン（配車作成・戻す・+イベント作成等）だけで
操作が完結するため、共通の下部固定ナビゲーションは置かない。

## ダークモード

**MVPでは実装しない**。将来的に必要になった時点で08_MVP開発計画
以降のフェーズで検討する。

## 「+ 車を追加」ボタン

一覧（8.2.1）の一番右に、通常の車カードと同じ幅の「＋」カードとして配置する。

## 未配車カードの表示位置

常に一番左に固定する。未配車の子供がいなくなれば自動的に非表示になる。

# 13. 検証・確認したいポイント（開発者自身によるセルフチェック項目）

配車担当への外部ヒアリングは行わない方針としたため、以下は開発者自身が
実装後に一通り操作して確認する項目とする。

- 「配車作成」ボタンを押してから配車案が出るまでの流れは分かりやすいか
- 定員超過・未配車の表示は目に留まりやすいか
- ドラッグ&ドロップでの修正は直感的か（2タップ方式の方が良いか）
- AIの判断理由（集合場所一致・経路上にある・定員に余裕あり・兄弟を同乗など）は
  納得感があるか、情報が多すぎないか
- 「+ 車を追加」カードは一覧の右端にあっても自然に見つけられるか
- LINE共有テキストの見やすさは現行のLINE投稿と比べてどうか
- タップ（AI理由表示）と長押し+ドラッグ（車移動）のジェスチャーが
  誤操作なく使い分けられるか
