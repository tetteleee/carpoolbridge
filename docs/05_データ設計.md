# 1. データ設計方針

MVPではシンプルなデータ構造を優先する。

- Firestoreを利用する
- 正規化しすぎない
- 入力しやすさを優先する
- 将来の拡張を考慮する

# 3. Family（家庭）

1家庭につき1レコード

| 項目 | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| id | String | ○ | ID |
| familyName | String | ○ | 〇〇家 |
| coachName | String \| null | | コーチ名。この家庭にコーチが紐づく場合のみ入力。空欄（null）はコーチなし |
| vehicleCapacity | Number | ○ | 車の総定員（運転者本人を含む。車を持たない場合は0） |
| pickupLocationId | String | ○ | 家庭の基本集合場所（子供の`pickupLocationOverride`がnullの場合に使用） |
| isActive | Boolean | ○ | 在籍中（falseで卒団・非表示扱い） |
| createdAt | Timestamp | ○ | |
| updatedAt | Timestamp | ○ | |

### coachName

この家庭の保護者がコーチを兼ねている場合のみ入力する（例：父親が保護者兼コーチ）。
保護者名の個別マスタは持たない。運転者・乗車メンバーとしての表示名は
familyNameから自動生成する（本章末尾および10章参照）。

イベントごとの参加要否は9章Responseの`coachParticipating`として
事前回答の対象に含め、自動配車の対象とする。車を出すかどうかは
専用項目を設けず、家庭の`driverOutward`/`driverReturn`（車出し）で
判断する（詳細は9章参照）。

コーチが車を出す場合（`coachParticipating`かつ`driverOutward`/`driverReturn`）、
そのコーチは運転者として10章のCarpool.membersに
`{ type: "coach", familyId, name: coachName, isDriver: true }`として含める
（乗車人数にもカウントする。定員自体が運転者を含む総定員として定義されているため、
9章参照）。
車を出さない場合は、所属家庭の子供と同じ集合場所から
乗客として自動配車の対象に含める（`isDriver: false`。子供が同じ
集合場所を利用する前提のため、所属家庭に子供が登録されていることを
前提とする。子供が登録されていない家庭に紐づくコーチは対象外とし、
10章の「その他の乗車者」として手動で追加する）。

※子供が在籍していない（家庭に紐づかない）コーチへの対応は、
本MVPでは扱わない。09_今後のアイデア.mdの将来検討事項とする。

### 運転者・保護者の表示名

保護者名の個別入力は行わない。車カード・LINE共有・乗車メンバー表示に
出る運転者名は、familyNameから「家」を除いた文字列（例：山田家→山田）を
そのまま使用する。コーチが運転する場合はcoachNameをそのまま表示する。

# 8. Response（イベント回答）

イベント配下に保存する。

1家庭につき1レコード。

## 家庭情報

| 項目 | 型 | 説明 |
| --- | --- | --- |
| familyId | String | |
| status | Enum | "未回答" または "回答済み" |
| driverOutward | Boolean | 行き車出し |
| driverReturn | Boolean | 帰り車出し |
| capacityToday | Number \| null | 当日乗車可能人数（運転者本人を含む総定員）の上書き。通常通り（未変更）ならnull |
| coachParticipating | Boolean \| null | コーチが参加するか。`Family.coachName`が空の家庭ではnull（入力対象外） |
| additionalParentNames | String[] | いつもは参加しないが今回同乗する保護者等の名前（複数可、任意） |
| remarks | String | 特記事項 |

### coachParticipating

`Family.coachName`が設定されている場合のみ入力対象となる項目。
コーチが紐づかない家庭では常にnullとする。

コーチが車を出すかどうかは、専用の項目を設けず`driverOutward`/`driverReturn`
（家庭の車出し）で判断する。1家庭が出す車は1台のみ（2台出すケースはない）
という前提のため、

- `coachParticipating = true` かつ `driverOutward = true` → そのコーチが運転者。10章のCarpool.membersに
`{ type: "coach", familyId, name: coachName, isDriver: true }`として含める
- `coachParticipating = true` かつ `driverOutward = false` → 所属家庭の子供と同じ集合場所から乗客として自動配車の対象に含める
（`isDriver: false`。帰りも同様に`driverReturn`で判断する）
- `coachParticipating = false` または null → 考慮しない

### additionalParentNames

いつもは配車の対象にならない保護者等（例：普段車を出さない父親が
今回だけ同乗する、祖父母が同乗する等）を、家庭単位で自由入力する。
1家庭につき複数人を登録できる（配列）。

個別マスタは持たず、名前のみを保持する。
配車対象への含め方はcoachParticipatingと同様の考え方に準拠する。

- その家庭が車を出す場合（`driverOutward`/`driverReturn`）：
運転者は別途Carpool.membersへ`{ type: "parent", isDriver: true }` として自動生成される（10章参照）ため、`additionalParentNames`の
人物は運転者とは別に、同乗する乗客として扱う
- いずれの場合も、所属家庭の子供と同じ集合場所から乗客として
自動配車の対象に含める（`isDriver: false`）
- 子供が登録されていない家庭の場合は対象外とし、10章の
「その他の乗車者」として手動で追加する

### status

"未回答" または "回答済み"。イベント作成直後の初期状態は "未回答"。
"未回答" の家庭に属する子どもは、自動配車の計算対象から除外され、配車画面では「未配車」リストに残される。

### capacityToday

当日の荷物等で車の総定員（`Family.vehicleCapacity`）から変動がある場合のみ数値を設定する。
未変更の場合は `null` とする。これにより「担当者が意図的に変更したのか」が明確になり、後日マスタの通常乗車人数が変更されてもイベント時点の状態が正確に残る。

## 子供情報

children[]

```json
[
  {
    "childId": "xxx",
    "isParticipating": true,
    "noOutwardRide": false,
    "noReturnRide": false,
    "remarks": ""
  },
  {
    "childId": "yyy",
    "isParticipating": true,
    "noOutwardRide": true,
    "noReturnRide": true,
    "remarks": "現地集合、帰りも別"
  }
]
```

| 項目 | 型 | 説明 |
| --- | --- | --- |
| childId | String | 子供ID |
| isParticipating | Boolean | イベントに参加するかどうか |
| noOutwardRide | Boolean | 行きの配車が不要かどうか（現地集合、午後から参加など） |
| noReturnRide | Boolean | 帰りの配車が不要かどうか（保護者迎え、現地解散など） |
| remarks | String | 特記事項（集合場所変更、同乗者ありなど） |

### isParticipating

イベント自体に参加するかどうかを表す。`false`（欠席）の場合は配車対象外となる。

### noOutwardRide

参加するが、行きの車での移動が不要な場合（現地集合や、別ルートでの遅れ参加など）に `true` とする。

### noReturnRide

参加するが、帰りの車での移動が不要な場合（保護者が直接迎えに来る、現地解散など）に `true` とする。

## 乗車メンバー

子供・保護者・コーチいずれも乗車しうるため、種別を持たせる。**運転者（保護者・コーチ問わず）は必ず`isDriver: true`のメンバーとして
membersに含める**。これにより「運転者は定員に含めるがmembersには
含めない」といった例外分岐を作らず、定員判定は常に `members.length > capacity`のみで行える（8章参照）。

保護者・コーチは、将来的に扱いを分けたくなる可能性を考慮し、
別々の`type`として保存する（現時点では表示色以外の挙動の違いはない）。
どちらも個人単位のマスタを持たないため、`familyId`＋`name`（8章の表示名ルールで
自動生成、コーチは`Family.coachName`）で表現する。

マスタに存在しない例外的な同乗者（祖父母、単発の助っ人など）に
対応するため、IDを参照せず名前を直接持つ`type: "other"`も用意する
（`other`は`isDriver`を持たない。運転者が`other`になるケースは
想定しない）。

```json
[
  { "type": "child", "id": "xxx" },
  { "type": "parent", "familyId": "family-001", "name": "山田", "isDriver": true },
  { "type": "parent", "familyId": "family-001", "name": "山田次郎の親（同乗のみ）", "isDriver": false },
  { "type": "coach", "familyId": "family-002", "name": "田中コーチ", "isDriver": false },
  { "type": "other", "name": "◯◯さん（祖母）" }
]
```

### type: "parent" について

- 運転する保護者：`driverOutward`/`driverReturn`が"可"の家庭について、
配車作成時に自動生成する（`isDriver: true`、nameは8章の表示名ルールで自動生成）。ただし、その家庭に
`coachParticipating: true`のコーチが紐づく場合は、運転者は
`type: "coach"`側で表現し、`type: "parent"`の運転者は生成しない
（1家庭1台のため、運転者は1人）
- 同乗のみの保護者：8章`Response.additionalParentNames`に基づき
生成する（`isDriver: false`）。複数人いる場合は名前ごとに
複数のメンバーとして追加する

### type: "coach" について

`coachParticipating: true`かつ家庭の車出しが可の場合、運転者として生成する
（`isDriver: true`、nameは`Family.coachName`）。車出しが不可の場合は
所属家庭の子供と同じ集合場所から乗客として生成する（`isDriver: false`）。

### type: "other" について

`child`・`parent`・`coach`と異なりマスタ登録を必要としない、自由入力の
乗車メンバー。配車担当が任意に追加・削除する。定員（乗車人数）にはカウントするが、
集合場所一致・兄弟同乗等のAI判断理由の算出対象にはしない
（そもそも参照する集合場所・家庭の情報を持たないため）。

# 11. 削除方針

論理削除を基本とする。

卒団した家庭・子供は物理削除せず、`isActive`をfalseにして非表示扱いとする。

- 新規イベントの回答画面・配車画面には表示しない
- 過去のイベント履歴（Response・Carpool）はfamilyId/childIdの参照先が
引き続き存在するため、名前を含め問題なく表示できる

物理削除は行わない。イベント履歴は約1年間保持する。
