# 1. データ設計方針

MVPではシンプルなデータ構造を優先する。

- Firestoreを利用する
- 正規化しすぎない
- 入力しやすさを優先する
- 将来の拡張を考慮する

# 5. Coach（コーチ）

コーチのみ乗車のケースに対応するため、保護者と同様にマスタ登録する。

|項目|型|必須|説明|
|---|---|---|---|
|id|String|○||
|name|String|○||
|isActive|Boolean|○|在籍中（falseで非表示扱い）|
|createdAt|Timestamp|○||
|updatedAt|Timestamp|○||

コーチのイベントごとの参加要否は事前回答の対象とせず、
配車担当がUC-06（配車修正）でCarpool.membersへ手動追加する運用とする。

## 子供情報

children[]

```json
[
  {
    "childId":"xxx",
    "outward":"join",
    "return":"carpool",
    "pickupLocationOverride":null
  },
  {
    "childId":"yyy",
    "outward":"local",
    "return":"pickup",
    "pickupLocationOverride":"locationId-zzz"
  }
]
```

### outward

- join
- absent
- local

### return

- carpool
- pickup
- local

### pickupLocationOverride

今回のみ通常と異なる集合場所を使う場合（例外「集合場所変更」）に設定する。
未指定（null）の場合はChild.pickupLocationIdを使用する。

# 12. 削除方針

論理削除を基本とする。

卒団した家庭・子供・コーチは物理削除せず、`isActive`をfalseにして非表示扱いとする。

- 新規イベントの回答画面・配車画面には表示しない
- 過去のイベント履歴（Response・Carpool）はfamilyId/childId/coachIdの参照先が
  引き続き存在するため、名前を含め問題なく表示できる

物理削除は行わない。イベント履歴は約1年間保持する。
