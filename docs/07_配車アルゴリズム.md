# 配車アルゴリズム

本ドキュメントは、学童野球配車支援システムにおける自動配車アルゴリズムの論理設計を定義する。
「人間の配車担当者が行う調整業務の80〜90%を自動化し、最終確認・手動修正を前提とする」というMVP方針に基づき、複雑な最適化や巡回順（ルーティング）算出は行わず、**「シンプルなスコアリングと貪欲法（Greedy）」**による設計を採用する。

---

## 1. 設計方針とシンプル化の論理的根拠

本アルゴリズムは、実装コストを最小化しつつ実用的な配車案を出力するため、以下の「引き算」の設計を行っている。

* **ルーティング機能の非搭載**:
  * 目的地までの最適巡回順の自動算出は行わない。実際の運用では、ドライバーが各自の使い慣れたナビアプリ等で最適ルートを選択するため、システム側では「同じ車に乗るメンバー（車ごとの集合場所リスト）」の確定に特化する。これにより、巡回順の反転補正ロジックやそれらに伴う保守コストを完全に排除する。
* **ドライバーの集合場所を起点とした距離評価**:
  * 経路全体との距離ではなく、「ドライバーの集合場所」と「グループの集合場所」の2点間距離のみを静的に評価する。これにより、割り当て状況によって評価関数が複雑化するのを防ぎ、テスト容易性（Testability）を最大化する。
* **厳格な事前バリデーション（Early Exit）**:
  * 緯度経度が未登録の地点や、優先割り当て時の定員超過などのデータ不整合がある場合、不完全なデータで強引に計算を進めるペナルティ処理（Soft Fail）は行わない。処理の先頭でエラー（Hard Fail）として検知・中断し、ユーザーへ修正を促すことで、不正データの伝播を防ぐ。

## 2. 前処理フェーズ（制約の事前解決とデータ構造化）

探索範囲を最小化し、業務上の絶対制約を事前に保証するため、メインループロジックの前にデータを以下の構造に変換する。
依存関係を正しく解決するため、バリデーションと構造化は以下の順序で厳密に実行する。

### 2.1 データの初期化と基本バリデーション
1. **緯度経度チェック（Early Exit）**:
   * 配車対象の全集合場所およびドライバーの集合場所において、緯度・経度が設定されているか検証する。未設定の地点が1つでもある場合は**自動配車処理をエラーで中断**し、エラー原因となった地点名を管理画面に通知する。
2. **車両データの初期化**:
   * 対象方向（行き/帰り）の車出しフラグが `true` である車両 $V$ を抽出する。
   * **有効定員 $remainingCapacity_V$ の算出**: 
     $$remainingCapacity_V = V.\text{capacity} - 1_{\text{driver}}$$
     （※ドライバーは乗車確定のため、あらかじめ定員から1を減算する）

### 2.2 家族グループ（Family Group）の形成
「兄弟は同じ車」「乗客コーチは所属家庭の子供と同じ集合場所から乗車」という絶対制約をコードの構造で解決するため、乗客を**家庭（`familyId`）単位で1つの「グループ $G$」**に統合する。

* **グループ $G$ のプロパティ（データ構造の統一）**:
  * `familyId`: 所属家庭ID
  * `size`: 必要席数（構成員の合計人数。ただしドライバー本人は除く）
  * `pickupLocationId`: 集合場所ID（永続化・同値比較用）
  * `pickupLocation`: 座標オブジェクト（アルゴリズムの距離計算用：緯度・経度）

### 2.3 優先割り当てグループの定員検証と割当
1. **定員超過チェック（Early Exit）**:
   * 車両 $V$ に強制的に割り当てられるドライバー家庭のグループ $G_{\text{driver}}$ （ドライバー本人を除いた、同乗必須の家族グループ）の必要席数 $G_{\text{driver}}.\text{size}$ が、車両の有効定員 $remainingCapacity_V$ を超過しているか検証する。
   * もし超過している車両が存在する場合、**自動配車処理を不整合データとして即時中断**し、対象ドライバー名をエラーとして通知する。
2. **ドライバー家族の優先割り当て**:
   * 各車両 $V$ に対応するグループ $G_{\text{driver}}$ を割り当て、車両の空き定員を更新する。
     $$remainingCapacity_V \leftarrow remainingCapacity_V - G_{\text{driver}}.\text{size}$$
   * 優先割り当て済みのグループは、以降の自動配車対象リスト（未配車グループ配列）から除外する。

## 3. スコアリング関数（配置適合度評価）

未割り当てのグループ $G$ を車両 $V$ に配置する際の適合度をシンプルに評価するため、以下のスコア関数を定義する。

$$\text{Score}(G, V) = \text{DISTANCE\_WEIGHT} \cdot \text{Distance}(G, V) + \text{STOP\_PENALTY\_WEIGHT} \cdot \text{IsNewStop}(G, V)$$

> **注意（評価値の性質と副作用の排除）**
> * 本スコアは物理的な単位（kmなど）を統一した絶対値ではなく、適合度の比較を目的とした**相対的な評価値（ヒューリスティックスコア）**である。スコア値が低いほど、配置適合度が高い（好ましい）と判定する。
> * 本スコアを計算する `calculateScore` は**副作用を一切持たない純粋関数（Pure Function）**として設計する。引数として渡されたグループや車両オブジェクトの状態を書き換える（破壊的変更を加える）ことは一切せず、与えられた情報から数値スコアを算出・返却する責任のみを負う。

### 定数定義（チューニングパラメータ）
重みの対称性および将来的なスケール調整を容易にするため、距離と経由地ペナルティそれぞれに係数を定義する。
* **`DISTANCE_WEIGHT`** $= 1.0$
  * 距離要素の重要度係数（基準値）。
* **`STOP_PENALTY_WEIGHT`** $= 5.0$
  * 車両の集合場所リストに新規経由地が追加される際のペナルティの重み。
  * *※本値はシステム共通の設定（環境変数や外部構成ファイル等）として管理し、**コード変更なしで調整可能**なパラメータとして扱う。*

### 各種パラメータ
#### 1. 距離評価： $\text{Distance}(G, V)$
* **定義**: ドライバー $V$ の集合場所（`driverPickupLocation`）と、グループ $G$ の集合場所（`pickupLocation`）の間のハバーシン公式による直線距離（km）。
* **意図**: 地域的に近いメンバー同士を同じ車両にまとめるための基本指標。

#### 2. 新規経由地判定： $\text{IsNewStop}(G, V)$
* **定義**: グループ $G$ の集合場所ID（`pickupLocationId`）が、すでに車両 $V$ の経由予定集合場所IDセット（`pickupLocationIds`）に含まれていない（新規に経由地が増える）場合は **`1.0`**、すでに経由予定に含まれている場合は **`0.0`** とする。
* **意図**: 同じ集合場所に集まるメンバーを可能な限り同一の車両にまとめる。

## 4. 割り当てフェーズ（貪欲法ロジック）

パッキング効率を高め、定員割れを極力防ぐため、人数の多いグループから順に割り当てる。

### 4.1 決定論的動作（再現性）の保証
実行環境やデータベースのレコード取得順に依存せず、常に100%同一の入力に対して同一の配車結果を出力するため、ソート条件および同点時の評価基準を以下のように厳密に定義する。

1. **配列順序の保護（Shallow Copyの適用）**:
   * メインロジック内でのソート処理が、引数として渡された元配列の並び順を破壊しないよう、車両配列およびグループ配列を複製（シャローコピー）した上でソート処理を行う。
   * *※なお、オブジェクトのプロパティ（membersやremainingCapacity等）に対する変更は、パフォーマンスおよびMVP設計のシンプルさを優先し、内部での参照ミューテーションを許容（呼出元への状態伝播を是認）する。*
2. **車両のソート優先度**:
   * 配車処理の前に、複製した車両配列を **`driverFamilyId` の昇順** で決定論的にソートする。
3. **グループ of ソート優先度**:
   * 複製した未配車グループ配列を、**第一順位：構成員数（`size`）の降順**、**第二順位：`familyId` の昇順** で決定論的にソートする。
4. **スコア同点時の優先順位**:
   * スコア計算において、複数の車両が同一スコアとなった場合は、**事前にソートされた車両配列のインデックスが若い車両を優先（先勝ち）**とする。疑似コード上では、適合条件を厳密な不等号 `score < minScore` で評価することで、同点時に後続の車両による上書きを防ぐ。

### 4.2 配車処理フロー（疑似コード）

```typescript
interface Location {
  latitude: number;
  longitude: number;
}

interface Passenger {
  id: string;
  name: string;
}

interface Group {
  familyId: string;
  size: number;
  pickupLocationId: string; // 永続化・比較用ID
  pickupLocation: Location; // アルゴリズム用座標オブジェクト
  members: Passenger[];
}

interface Vehicle {
  id: string;
  driverFamilyId: string;
  remainingCapacity: number;
  driverPickupLocationId: string; // 永続化・比較用ID
  driverPickupLocation: Location; // アルゴリズム用座標オブジェクト
  pickupLocationIds: Set<string>; // 経由予定の集合場所IDセット（O(1)検索用）
  members: Passenger[];
}

const DISTANCE_WEIGHT = 1.0;
const STOP_PENALTY_WEIGHT = 5.0;

function runAutoAssignment(
  vehicles: Vehicle[],
  unassignedGroups: Group[]
): { assignedVehicles: Vehicle[]; unassignedList: Group[] } {

  // 1. 車両配列のコピーと決定論的ソート（ドライバー家庭IDの昇順）
  const sortedVehicles: Vehicle[] = [...vehicles].sort((a, b) => 
    a.driverFamilyId.localeCompare(b.driverFamilyId)
  );

  // 2. 未配車グループ配列のコピーと決定論的ソート（第一順位: 構成員数の降順、第二順位: 家庭IDの昇順）
  const sortedGroups: Group[] = [...unassignedGroups].sort((a, b) => {
    if (b.size !== a.size) {
      return b.size - a.size;
    }
    return a.familyId.localeCompare(b.familyId);
  });

  const unassignedList: Group[] = [];

  for (const group of sortedGroups) {
    let bestVehicle: Vehicle | null = null;
    let minScore = Infinity;

    // 3. このグループが乗車可能な車両の中から、最もスコアが低い車両を探索
    for (const vehicle of sortedVehicles) {
      if (vehicle.remainingCapacity >= group.size) {
        const score = calculateScore(group, vehicle);
        // 同点時は厳格不等号により、インデックスの若いソート優先車両が勝ち残る
        if (score < minScore) {
          minScore = score;
          bestVehicle = vehicle;
        }
      }
    }

    // 4. 割り当ての実行、または未配車リストへの退避（ミューテーションを許容）
    if (bestVehicle) {
      bestVehicle.members.push(...group.members);
      // 有効定員の更新: remainingCapacity_V ← remainingCapacity_V - G.size
      bestVehicle.remainingCapacity -= group.size;
      bestVehicle.pickupLocationIds.add(group.pickupLocationId);
    } else {
      unassignedList.push(group);
    }
  }

  return { assignedVehicles: sortedVehicles, unassignedList };
}

// 適合スコア計算（副作用を持たない純粋関数）
function calculateScore(group: Group, vehicle: Vehicle): number {
  const distance = getHaversineDistance(vehicle.driverPickupLocation, group.pickupLocation); // 直線距離(km)
  const isNewStop = vehicle.pickupLocationIds.has(group.pickupLocationId) ? 0.0 : 1.0;
  
  return (DISTANCE_WEIGHT * distance) + (STOP_PENALTY_WEIGHT * isNewStop);
}
```

## 5. 計算量（Complexity）

実用スケール（乗客数 $N \le 40$、車両数 $M \le 10$）を前提とした計算効率の評価。

* **ソート**: グループ数を $N_G$ （家族単位に統合されているため $N_G \le N$）とすると、 $O(N_G \log N_G)$。車両ソートは $O(M \log M)$。
* **マッチング**: 各グループについて全車両を評価するため、最悪ケースでも $O(N_G \times M)$。
  * **スコア計算におけるハッシュ探索効率**: `calculateScore` 内で行われる経由地判定（`vehicle.pickupLocationIds.has(...)`）は、JavaScriptの内部データ構造 `Set` （ハッシュテーブル）を用いて評価される。ハッシュ衝突を考慮しない平均計算量は **$O(1)$** であるため、スコア計算処理自体は定数時間 $O(1)$ で完了し、マッチング全体のボトルネックとはならない。
* **総合計算量**: 
  $$O(N_G \log N_G + M \log M + N_G \cdot M)$$
  
本システムで想定するデータ量においては、クライアントサイド（モバイル端末を含む）での処理であっても、十分高速に処理が完了するため、非機能要件を完全に満たす。

## 6. 例外系・境界条件設計

| イレギュラー状態 | アルゴリズム上の挙動 | UI/UXへの提示方法 |
| :--- | :--- | :--- |
| **車両の総空き定員が不足** | 溢れたグループを `unassignedList` に格納して処理を完了する。 | 「配車枠が不足しています。車出しを追加するか、未配車メンバーを手動で車へ割り当ててください」と警告表示し、未配車エリアにメンバーを一覧表示する。 |
| **緯度経度情報の未登録** | 計算を開始せず、早期にエラー（Hard Fail）とする。 | 「位置情報が未登録の集合場所（[集合場所名]）が存在するため、自動配車を開始できません」と明確なエラーを表示する。 |
| **優先割り当てグループが定員を超過** | 計算を開始せず、早期に不整合データエラー（Hard Fail）とする。 | 「[ドライバー名]様の優先割り当て人数（同乗必須メンバー数）が、車両の有効定員（車両定員 - 1名）を超過しています。登録データを確認してください」と警告表示する。 |
| **イベント未回答者** | 自動配車の入力データから除外する。 | 「未回答者が $X$ 名います」とシステム警告を出し、手動調整用の未配車リストに初期配置する。 |
| **直前の急なメンバー追加/欠席** | 自動配車は常に「最新の回答状態」から一から綺麗な配車案を高速に再計算する。 | 自動再生成を行うと既存の手動調整がリセットされる旨の確認ダイアログを表示する。または、UI上のドラッグ＆ドロップによる手動調整を優先させる。 |

## 7. アルゴリズムの限界と運用上の位置づけ

本アルゴリズムは、人間の配車担当者が行う調整の大部分を自動化し、初期配車案を高速に提示することを目的として設計されている。

* **決定論的アルゴリズムとしての再現性**:
  本アルゴリズムは決定論的（Deterministic）に動作するよう厳密に設計されており、**同一の入力データセットに対しては常に100%同一の結果を出力する（高い再現性の保証）**。
* **最適解の非保証**:
  貪欲法（Greedy法）による局所的かつ段階的な割り当て処理を採用しているため、**数学的な「全体最適解」を保証するものではない。**
  実運用上において「この配車案よりも、さらに効率的な組み合わせが存在する」状況は理論上発生し得るが、本システムは「自動配車後にユーザー自身がUI上でドラッグ＆ドロップにより手動微調整を行うこと」を前提のフローとしている。したがって、極端なケースでの非効率性はバグではなく仕様の範疇（許容される限界値）として扱い、過度な最適化ロジックの追加による複雑化を回避する。
