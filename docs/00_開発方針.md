# AI開発方針

Version: 3.2

---

# 1. 目的

本プロジェクトはAIを活用し、短時間でも継続して開発できることを目的とする。

目標は

> **「配車担当が毎週使いたくなるアプリ」を完成させること**

である。

コードの美しさよりも、

- 現場で使いやすいこと
- 継続して開発できること

を優先する。

---

# 2. 基本方針

- ドキュメントを唯一の仕様とする
- Taskを単位として開発する
- AIは実装支援ツールとして利用する
- 最終的な設計判断は人間が行う
- MVPを最優先する
- 完璧より継続を優先する

---

# 3. AIの役割

## ChatGPT

役割

設計・技術相談

担当

- 要件整理
- 設計レビュー
- Task作成
- React設計
- Firebase設計
- TypeScript設計
- 将来の拡張性レビュー
- 実装方針の相談

---

## 実装ツール（AI IDE）

役割

実装担当

使用ツール

- Google Antigravity IDE（メイン）
- VSCode（必要に応じて）
- Claude Code（導入予定）

各ツールの無料枠を活用しながら実装を進めるため、
Task単位で使用ツールを使い分けることがある。

担当

- Taskに沿ったコード生成
- UI実装
- 軽微なリファクタリング
- バグ修正

実装ツールには仕様を考えさせない。

Task以外の独自判断は極力行わせない。

どのツールを使う場合でも、渡すTask・参照ドキュメントの内容は
同一とする（使用ツールが変わっても仕様は変わらない）。

---

## Claude

役割

レビュー・設計相談

担当

- コードレビュー
- React Best Practice
- TypeScript Best Practice
- 保守性
- 可読性
- 必要に応じた設計レビュー

Claudeはレビューだけでなく、設計相談にも利用する。

---

# 4. AIへの指示原則

AIへ依頼する際は、以下を徹底する。

- Taskを必ず渡す
- 参照ドキュメントを渡す
- スコープ外を明確に伝える
- 一度に複数Taskを依頼しない
- 完了条件を満たしたら実装を終了する
- 勝手な機能追加をさせない
- 勝手な設計変更をさせない
- 分からない仕様は推測させず質問させる

AIは仕様を決める存在ではなく、

**仕様を実装する存在**

として扱う。

これは実装ツール（Google Antigravity IDE / VSCode / Claude Code）を
どれに切り替えても共通のルールとする。

# 5. 開発フロー

## 通常Task

```
Task作成
      ↓
実装ツールで実装
      ↓
npm run build
      ↓
自分で動作確認
      ↓
Git Commit
```

---

## 重要Task

対象

- 配車画面
- 配車アルゴリズム
- Firestore設計
- 認証・権限
- 大規模リファクタリング

```
Task作成
      ↓
実装ツールで実装
      ↓
npm run build
      ↓
自分で動作確認
      ↓
Claudeレビュー
      ↓
Criticalあり？
      ├── Yes → 実装ツールで修正 → Claude再レビュー
      └── No
              ↓
必要ならChatGPT相談
              ↓
Git Commit
```

毎Taskで複数AIを往復させない。

レビューは重要Taskのみ実施する。

# 6. Task管理

Taskは半日〜1日で終わるサイズに分割する。

Taskには必ず以下を記載する。

- 目的
- 完了条件
- 実装内容
- スコープ外
- 確認項目
- 参照ドキュメント

例

```
参照ドキュメント

01_要件定義
・5章

03_データ設計
・Families

04_プロトタイプ
・7章
```

使用する実装ツール名（Antigravity / VSCode / Claude Code）は
Task自体には記載不要とする。Taskの内容はツールに依存しない
共通の仕様として書く。

# 8. 動作確認

各Taskには最低限の確認項目を書く。

例

```
□ npm run build が成功する

□ エラーが発生しない

□ スマホ表示が崩れない

□ 完了条件を満たしている
```

重要機能については手動確認を行う。

対象

- イベント作成
- 回答入力
- 配車作成
- ドラッグ&ドロップ
- LINE共有
- 認証・権限

認証・権限Taskでは以下も確認する。

```
□ 未登録UIDではアクセスできない

□ 登録済みUIDでは利用できる

□ Firestore Rulesが期待通り動作する
```

# 9. Git運用

Task単位でコミットする。

ブランチ例

```
feature/task001-router

feature/task004-dispatch-ui
```

コミット例

```
feat: Task001 Router追加

feat: Task004 配車画面追加

fix: Task004 Claudeレビュー対応
```

使用した実装ツール名はコミットメッセージに含めなくてよい
（Task番号でどのTaskの成果か追跡できれば十分なため）。

重要Taskでレビュー指摘（Critical/Major）があった場合は、
別ファイルに残さず、修正コミットのメッセージに指摘内容を含める。

```
fix: Task004 Claudeレビュー対応

Critical: Firestore Rulesの家庭チェックが漏れていた点を修正
Major: useEffectの依存配列にeventIdが抜けていた点を修正
```

`git log --grep=Critical` 等で後から検索できるため、
reviews/ディレクトリでのファイル管理は行わない。
指摘がなかった（Good）Taskについては何も残さなくてよい。

# 11. 開発の優先順位

優先順位は以下とする。

1. 動くこと
2. 現場で使いやすいUI
3. 保守性
4. リファクタリング
5. パフォーマンス最適化

MVP段階では過度な最適化・汎用化は行わない。

---

# 12. 成功条件

成功とは、

> **「配車担当が毎週このアプリを使いたい」と思えること**

である。

迷ったときは、

- 現場で使いやすいか
- MVPとして十分か
- 継続して開発できるか

を優先して判断する。

---

# 13. この方針書の位置付け

本ドキュメントは固定されたルールではない。

実際にTaskを進める中で、

- 開発効率
- AIの使い分け
- 実装ツールの選定
- レビュー方法

などを見直し、継続的に改善する。

「運用しながら育てるドキュメント」として扱う。
