# AI活用ガイド

## 目的

このプロジェクトは、**できるだけ無料でAIを活用しながら配車アプリを完成させること**を目的とする。

AIの利用量には限りがあるため、

- トークン消費を抑える
- やり直しを減らす
- MVP（必要最低限の機能）を最優先にする

ことを常に意識する。

---

# 基本方針

## MVPを最優先とする

最初から完成形を目指さない。

まずは、

> **「配車が成立すること」**

だけを目標とする。

MVP完成後に機能追加していく。

---

## 機能追加方式で開発する

最初に実装する機能例

- イベント作成
- イベント一覧
- 参加者入力
- 配車実行
- LINE共有

その後に追加する機能例

- 回答済み表示
- 未回答表示
- 検索
- 並び替え
- 通知
- 管理者追加
- お気に入り
- UI改善

「今なくても配車できる機能」は後回しにする。

# AIを節約するためのルール

## 1. 一度に大きな変更を依頼しない

悪い例

> イベント画面を全部作り直して

良い例

- 回答ボタンを右上へ移動
- 日時を追加
- アイコンを変更

変更はできるだけ小さく分割する。

---

## 2. 実装も小さく分ける

悪い例

> イベント機能を全部実装して

良い例

1. Firestoreへ保存
2. 一覧表示
3. 編集
4. 削除

1機能ずつ完成させる。

---

## 3. ドキュメントを最新に保つ

PROJECT.mdおよびdocs配下を常に最新に保つ。

設計資料が整っていれば、毎回AIへ長文説明する必要がなくなる。

---

## 4. PROJECT.mdを参照させる

毎回アプリの説明を書かない。

「PROJECT.mdを参照してください」

で済む状態を維持する。

---

## 5. リファクタリングは最後

コードが多少汚くても、

- 動く
- 読める

のであれば後回し。

途中で何度も整理するとAI利用量が増える。

---

## 6. UIは最低限で良い

最初は

- ボタン
- 入力欄
- リスト

だけで十分。

以下はMVP完成後でよい。

- アニメーション
- 細かな余白
- 装飾

---

## 7. Gitコミットを細かく行う

機能単位でコミットする。

例

- イベント作成
- イベント編集
- 配車ロジック
- LINE共有

AIが大きく壊してもすぐ戻せる。

# 推奨する開発フロー

```text
ChatGPT
    │
    │ 要件整理・仕様相談
    ▼
Claude
    │
    │ 設計資料更新
    ▼
Claude Code, GitHub Copilot
    │
    │ 1機能だけ実装
    ▼
動作確認
    ▼
Gitコミット
    ▼
ChatGPT
    │
    │ レビュー・改善点確認
    ▼
次の機能へ
```

# アイデア管理

開発中は新しいアイデアが必ず増える。

思いついてもすぐ実装しない。

すべて

**09_今後のアイデア.md**

へ記録する。

MVP完成までは原則追加しない。

---

## 判断基準

新しい機能を追加したくなったら、必ず次の質問をする。

> **この機能が無いと配車できないか？**

- YES → MVPへ含める
- NO → 今後のアイデアへ記録

---

# 最重要ルール

**完成を優先する。**

途中で仕様を広げない。

途中でUIにこだわらない。

途中でコードを綺麗にし過ぎない。

**まず動くものを作り、完成後に改善する。**
